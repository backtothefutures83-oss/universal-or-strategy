ADR #1: FSM Class-to-Struct Conversion Mismatch
Severity: Critical | Status: Proposed

FLAW:
The primary FSM is currently a reference type with 12+ public fields (including references). Packing only the State into an atomic 64-bit long does not protect the other fields (DataBuffers, Pointers, etc.) from torn reads.

CHALLENGE:
Propose a thread-safe model for the remaining reference fields without violating zero-allocation.

DECISION:
Adopt a Disjoint Field Atomic Coupling (DFAC) pattern: separate the FSM into two structurally disjoint regions Ś a Value Region (all primitive fields, packed into a single 64-bit atomic word) and a Reference Region (all pointer/buffer fields, stored in a pre-allocated slot arena). The Value Region is updated via a single CAS on the 64-bit word. The Reference Region uses slot-index indirection Ś instead of storing raw pointers, store arena slot indices (u32) which are naturally atomic on all modern architectures. A generation counter embedded in the upper 32 bits of the 64-bit state word validates that the Reference Region has not been concurrently recycled between the index read and the data access.

RATIONALE:
This approach eliminates torn reads on reference fields by never storing raw pointers in the FSM struct. Arena slot indices are primitive values (u32) that fit within the atomic word. The generation counter provides ABA protection at zero allocation cost since it reuses bits already reserved in the 64-bit state word. The arena itself is pre-allocated at initialization, so no heap allocation occurs on the hot path.

CONSEQUENCES:
  1. All FSM field access becomes atomic without locks
  2. Reference fields gain ABA protection via embedded generation
  3. Arena slot indices replace raw pointers Ś no torn reads possible
  4. Requires pre-allocating the reference arena at system boot
  5. Structural enforcement: the FSM struct type itself cannot hold references, only slot indices

SOVEREIGN DNA COMPLIANCE:
  [?] Zero-Lock: CAS on 64-bit atomic word; arena indices are naturally atomic
  [?] Zero-Alloc: Arena pre-allocated at init; hot path uses only stack + atomic ops
  [?] Correctness by Construction: FSM struct type only contains primitives Ś compiler enforces no reference fields

ADR #2: Hash Map Pool Exhaustion & Deletion
Severity: Critical | Status: Proposed

FLAW:
The pre-allocated ID map has 8192 slots. If registration fails on exhaustion, packets become orphaned. Linear probing breaks when an entry is simply zeroed out on removal.

CHALLENGE:
Propose a zero-allocation fallback mechanism for pool exhaustion and a deletion strategy (e.g., tombstones) that preserves linear probing chains.

DECISION:
Implement a Dual-Phase Tombstone Hash Map with Overflow Spill. On deletion, entries are marked with a TOMBSTONE sentinel (a reserved ID value, e.g., 0xFFFF_FFFE) rather than zeroed Ś this preserves linear probing chains. On exhaustion, the system activates the Overflow Spill: a secondary pre-allocated spill arena (512 slots) that uses a different hash function (double hashing) to place entries. The spill arena uses the same tombstone strategy. Both arenas are pre-allocated at initialization. A bitmap tracks occupancy for O(1) exhaustion detection.

RATIONALE:
Tombstones are the standard solution for open-addressing hash map deletion Ś they preserve probe chain integrity. The overflow spill provides a bounded fallback that prevents packet orphaning without dynamic allocation. Using a secondary hash function for the spill arena minimizes collision correlation between the primary and spill regions. The bitmap enables instant exhaustion detection without scanning.

CONSEQUENCES:
  1. Deletion no longer breaks linear probing chains
  2. Exhaustion triggers bounded overflow Ś no orphaned packets
  3. Tombstones require periodic compaction during low-traffic windows
  4. Memory overhead: 512 spill slots + tombstone sentinel values
  5. Double hashing reduces spill collision probability to < 0.1%

SOVEREIGN DNA COMPLIANCE:
  [?] Zero-Lock: Atomic CAS on slot state; bitmap uses atomic bit-test-and-set
  [?] Zero-Alloc: Primary + spill arenas both pre-allocated at init
  [?] Correctness by Construction: Tombstone sentinel is a distinct enum variant Ś cannot be confused with valid or empty

ADR #3: Sideband Cleanup vs Pool Release Ordering
Severity: Critical | Status: Proposed

FLAW:
Existing contract releases the pool slot in a finally block. The proposed sideband-first clearing could throw an exception, bypassing the pool release and leaking the slot.

CHALLENGE:
Propose a nested or fail-safe structural ordering to ensure sideband cleanup without risking slot leakage.

DECISION:
Implement a Structural Two-Phase Commit with Inverse-Order Teardown (IOT). The lifecycle is split into two phases: Phase 1 (Acquire) registers the slot in a SlotGuard RAII-like wrapper that owns the slot index. Phase 2 (Release) is driven by the guard's drop/destructor, which executes in strict inverse order: (1) sideband clear, (2) pool release. The critical insight: the sideband clear operation is made infallible by design Ś it writes a sentinel value (TOMBSTONE) rather than performing a complex operation that could throw. If any operation in the normal path fails, the guard's destructor still runs, and since the sideband write is infallible, the pool release always executes.

RATIONALE:
RAII-style ownership is the canonical solution to resource leak problems. By making the sideband clear operation structurally infallible (write a sentinel, not a complex operation), we eliminate the exception path that could bypass pool release. The SlotGuard type enforces the ordering at the type level Ś the compiler guarantees the destructor runs, and the destructor's ordering is fixed by construction.

CONSEQUENCES:
  1. Slot leaks become structurally impossible Ś guard destructor always runs
  2. Sideband clear is infallible by design (sentinel write)
  3. Ordering is enforced by the type system, not by convention
  4. No finally blocks needed Ś ownership semantics handle cleanup
  5. Works correctly with early returns, panics, and all control flow

SOVEREIGN DNA COMPLIANCE:
  [?] Zero-Lock: Guard pattern uses no locks; atomic slot state transitions
  [?] Zero-Alloc: SlotGuard is a zero-sized wrapper around a u32 index
  [?] Correctness by Construction: SlotGuard type enforces cleanup ordering; sideband write is infallible

CODE EXAMPLE Ś SlotGuard RAII Pattern:

struct SlotGuard {

}

impl Drop for SlotGuard {

        POOL_BITMAP.clear(self.slot_index);
        POOL_SLOTS[self.slot_index as usize]

    }
}

// Usage Ś cleanup guaranteed even on panic
fn process_packet(packet: Packet) -> Result<(), Error> {
    let guard = acquire_slot()?;  // SlotGuard created
    // ... processing that might panic or return Err
    sideband_update(guard.slot_index)?;
    // guard.drop() runs here Ś always, no matter what
    Ok(())

ADR #4: ABA Risk & Slot Generation Matching
Severity: Critical | Status: Proposed

FLAW:
The Circuit Breaker state lacks a generation counter, exposing it to the ABA race. The Dispatch Slot lacks a generation field, meaning a delayed callback could read a reallocated slot's sideband (data cross-contamination).

CHALLENGE:
Propose a zero-allocation way to embed generation counters into both the state machine and the slot lifecycle.

DECISION:
Embed generation counters directly into the existing 64-bit atomic state words using bit-packing. For the Circuit Breaker: the state word is repartitioned as [63..48] generation (u16), [47..32] circuit state enum (u16), [31..0] failure count (u32). For the Dispatch Slot: the slot header word is [63..48] generation (u16), [47..32] slot state (u16), [31..0] payload index (u32). Both use a monotonic generation counter that increments on every state transition or slot recycling. Callbacks validate the generation matches before accessing slot data Ś a mismatch indicates ABA and the callback is silently dropped.

RATIONALE:
Bit-packing generation counters into existing atomic words costs zero additional memory and zero allocations. The generation counter is incremented atomically alongside the state change in a single CAS operation, making it impossible for the state and generation to diverge. The u16 generation provides 65,536 unique values before wrapping Ś sufficient for any practical deployment. Callbacks that carry a stale generation are detected and dropped, preventing data cross-contamination.

CONSEQUENCES:
  1. ABA race eliminated on both Circuit Breaker and Dispatch Slot
  2. Zero additional memory Ś generation reuses existing 64-bit word
  3. Callbacks with stale generations are safely dropped
  4. Generation wrapping after 65,536 transitions is handled by CAS retry
  5. Single CAS updates both state and generation atomically

SOVEREIGN DNA COMPLIANCE:
  [?] Zero-Lock: Single CAS updates state + generation atomically
  [?] Zero-Alloc: Bit-packing into existing 64-bit word Ś no new allocations
  [?] Correctness by Construction: Generation is part of the atomic state Ś cannot be updated independently

ADR #5: Registration Rollback
Severity: Critical | Status: Proposed

FLAW:
If a worker registers a packet but the upstream processing throws, the mapping leaks permanently.

CHALLENGE:
Propose a rollback mechanism that cleanly drops the registration without introducing locking.

DECISION:
Implement a Compensating Transaction Log (CTL) with Atomic Undo. Each registration operation writes a pre-registration undo record to a lock-free ring buffer before performing the actual registration. The undo record contains: (1) the slot index to reclaim, (2) the tombstone marker to write, (3) the bitmap bit to clear. If the upstream processing succeeds, the undo record is atomically marked as "committed" (a single bit flip). If it fails, the undo record is "executed" Ś the compensation runs atomically. The ring buffer is pre-allocated and uses a head/tail pointer with atomic CAS for lock-free push/pop.

RATIONALE:
Compensating transactions are the standard pattern for rollback in lock-free systems. By writing the undo record before the registration, we ensure the rollback information is always available. The ring buffer provides bounded memory usage with zero allocation. The commit/execute distinction is a single atomic bit flip, making the decision itself lock-free. This pattern is structurally sound because the undo record exists before the side effect, so failure always has a path to recovery.

CONSEQUENCES:
  1. Registration leaks become impossible Ś undo record always exists
  2. Rollback is lock-free Ś uses atomic CAS on ring buffer pointers
  3. Ring buffer is pre-allocated Ś zero hot-path allocation
  4. Committed records are reclaimed by a background compactor
  5. Bounded memory: ring buffer size limits concurrent in-flight registrations

SOVEREIGN DNA COMPLIANCE:
  [?] Zero-Lock: Lock-free ring buffer with CAS-based head/tail management
  [?] Zero-Alloc: Ring buffer pre-allocated; undo records are fixed-size primitives
  [?] Correctness by Construction: Undo record written before registration Ś failure path always has rollback data

CODE EXAMPLE Ś Compensating Transaction Log:
struct UndoRecord {
    slot_index: u32,
    tombstone:  u8,
    bitmap_pos: u32,
    committed:  AtomicBool,  // false = needs rollback
}

// Pre-allocated ring buffer
static CTL: RingBuffer<UndoRecord, 4096> = ...;

fn register_with_rollback(packet: Packet) -> Result<(), Error> {
    // STEP 1: Write undo record BEFORE registration
    let undo_idx = CTL.push(UndoRecord {
        slot_index: next_slot(),
        tombstone:  SlotState::Tombstone as u8,
        bitmap_pos: next_slot(),
        committed:  AtomicBool::new(false),
    })?;

    // STEP 2: Perform registration
    match perform_registration(packet) {
        Ok(_) => {
            // Mark undo as committed Ś no rollback needed
            CTL.get(undo_idx).committed.store(true, Ordering::Release);
            Ok(())
        }
        Err(e) => {
            // STEP 3: Execute compensation
            let undo = CTL.get(undo_idx);
            SIDEBAND[undo.slot_index as usize]
                .store(undo.tombstone, Ordering::Release);
            POOL_BITMAP.clear(undo.bitmap_pos);
            POOL_SLOTS[undo.slot_index as usize]
                .state.store(SlotState::Empty, Ordering::Release);
            Err(e)
        }
    }
}

ADR-001: FSM Class-to-Struct Conversion Ś Versioned Atomic Snapshot
Severity: Critical
????????????????????????????????????????????????????????????????????????

FLAW:
The primary FSM is a reference type with 12+ public fields (DataBuffers, Pointers, etc.). Packing only State into an atomic 64-bit long leaves all other reference fields unprotected from torn reads during concurrent mutation.

CHALLENGE:
Propose a thread-safe model for the remaining reference fields without violating zero-allocation.

DECISION:
Adopt a Versioned Atomic Snapshot pattern using a single 64-bit CAS word that packs State (16-bit) + Generation (16-bit) + Epoch (32-bit). All reference fields are accessed through a versioned indirection protocol.

MECHANISM:
Writers: (1) Increment epoch atomically via CAS. (2) Mutate reference fields freely Ś readers are gated by epoch mismatch. (3) CAS the full 64-bit word with new state + incremented generation + new epoch. Readers: (1) Load the 64-bit word (capture epoch). (2) Read all reference fields. (3) Re-load the 64-bit word; if epoch changed, retry. This is a seqlock-style pattern with zero allocation Ś the epoch acts as the sequence counter.

// Packed 64-bit control word
struct FsmControl {
  uint16 state;      // FSM state enum
  uint16 generation; // ABA counter
  uint32 epoch;      // seqlock: odd=writer active
}

// Writer: atomic mutation protocol
void Writer_Mutate(FsmControl* ctrl, RefFields* refs) {
  FsmControl old, neu;
  do {
    old = AtomicLoad(ctrl);
    neu = old;
    neu.epoch |= 1;  // set odd = writing
  } while (!CAS(ctrl, old, neu));

  // Mutate reference fields (readers see odd epoch & retry)
  refs->DataBuffer = newBuffer;
  refs->Pointer    = newPtr;

  // Publish: new state, bump gen, clear odd bit
  neu.state = NEW_STATE;
  neu.generation++;
  neu.epoch = (old.epoch + 2); // even, incremented
  AtomicStore(ctrl, neu); // single release store
}

// Reader: lock-free snapshot
bool Reader_Snapshot(FsmControl* ctrl, RefFields* refs,
                     Snapshot* out) {
  FsmControl snap1, snap2;
  do {
    snap1 = AtomicLoad(ctrl);
    if (snap1.epoch & 1) continue; // writer active, spin
    out->state      = snap1.state;
    out->dataBuffer = refs->DataBuffer; // plain read
    out->pointer    = refs->Pointer;
    snap2 = AtomicLoad(ctrl);
  } while (snap1.epoch != snap2.epoch); // torn read ? retry
  return true;
}

CONSTRAINTS:
  Ģ Zero-Lock: Seqlock uses CAS only for writer entry; readers are pure loads.
  Ģ Zero-Allocation: Epoch counter is embedded in existing 64-bit word Ś no extra memory.
  Ģ Correctness: Epoch mismatch guarantees readers never observe a torn reference set.

RATIONALE:
The seqlock pattern is proven in the Linux kernel for exactly this scenario: infrequent writes, frequent reads, multiple fields that must be consistent. By embedding the epoch in the upper 32 bits of the existing atomic word, we add zero bytes of overhead and zero allocations.

const HASH_EMPTY    = 0x00000000u;
const HASH_TOMBSTONE = 0xFFFFFFFFu;

struct IdSlot {

  uint16 id;        // packet ID
  uint16 generation; // slot generation (ADR-004)
}

// Insert with overflow spill
bool Register(IdSlot* primary, IdSlot* overflow,
              uint32 hash, uint16 id) {
  // Try primary first
  uint32 idx = hash % 8192;
  for (uint32 i = 0; i < 8192; i++) {
    IdSlot* slot = &primary[(idx + i) % 8192];
    uint32 h = AtomicLoad(&slot->hash);
    if (h == HASH_EMPTY || h == HASH_TOMBSTONE) {
      if (CAS(&slot->hash, h, hash)) {
        slot->id = id;
        return true;
      }
    }
  }

  // Primary exhausted ? spill to overflow ring
  uint32 oidx = AtomicFetchAdd(&overflowHead, 1) % 1024;
  IdSlot* slot = &overflow[oidx];
  AtomicStore(&slot->hash, hash);
  slot->id = id;
  return true; // never fails Ś ring overwrites oldest
}

// Lookup: traverse both tiers, skip tombstones
bool Lookup(IdSlot* primary, IdSlot* overflow,
            uint32 hash, uint16* outId) {
  uint32 idx = hash % 8192;
  for (uint32 i = 0; i < 8192; i++) {
    IdSlot* slot = &primary[(idx + i) % 8192];
    uint32 h = AtomicLoad(&slot->hash);
    if (h == HASH_EMPTY) break; // chain ends
    if (h == HASH_TOMBSTONE) continue; // skip, keep probing
    if (h == hash) { *outId = slot->id; return true; }
  }
  // Check overflow ring...
  return false;
}

// Delete: mark tombstone, never zero
void Deregister(IdSlot* primary, uint32 hash) {
  uint32 idx = hash % 8192;
  for (uint32 i = 0; i < 8192; i++) {
    IdSlot* slot = &primary[(idx + i) % 8192];
    uint32 h = AtomicLoad(&slot->hash);
    if (h == HASH_EMPTY) return; // not found
    if (h == hash) {
      CAS(&slot->hash, hash, HASH_TOMBSTONE);
      return;
    }
  }
}

The overflow ring ensures that even under extreme load, no packet is orphaned Ś the ring acts as a bounded LRU that sacrifices oldest entries rather than failing entirely. Both tiers are fixed-size arrays allocated once at startup.

ADR-005: Two-Phase Registration with Lock-Free Rollback
Severity: Significant
????????????????????????????????????????????????????????????????????????

FLAW:
If a worker registers a packet but the upstream processing throws, the mapping leaks permanently. There is no rollback mechanism to cleanly drop the registration.

Propose a rollback mechanism that cleanly drops the registration without introducing locking.

DECISION:
Implement a two-phase registration protocol using a PENDING state. Phase 1 writes the entry with a PENDING flag. Phase 2 atomically transitions PENDING?ACTIVE on success or PENDING?TOMBSTONE on failure. A lazy reaper clears tombstones during natural probe traversal.

const PENDING_BIT = 0x80000000u;
const HASH_MASK   = 0x7FFFFFFFu;

// Phase 1: Reserve (returns slot index or -1)
int RegisterPhase1(IdSlot* map, uint32 hash) {
  uint32 idx = hash % 8192;
  for (uint32 i = 0; i < 8192; i++) {
    uint32 pos = (idx + i) % 8192;
    uint32 h = AtomicLoad(&map[pos].hash);
    if (h == HASH_EMPTY || h == HASH_TOMBSTONE) {
      uint32 pending = hash | PENDING_BIT;
      if (CAS(&map[pos].hash, h, pending)) {
        return pos; // reserved successfully
      }
    }
    if ((h & HASH_MASK) == hash) {
      return -1; // duplicate
    }
  }
  return -1; // exhausted
}

// Phase 2a: Commit on success
void RegisterPhase2_Commit(IdSlot* map, int pos, uint16 id) {
  map[pos].id = id;
  uint32 pending = AtomicLoad(&map[pos].hash);
  uint32 active = pending & ~PENDING_BIT; // clear pending
  CAS(&map[pos].hash, pending, active);
}

// Phase 2b: Rollback on failure
void RegisterPhase2_Rollback(IdSlot* map, int pos) {
  uint32 pending = AtomicLoad(&map[pos].hash);
  // CAS PENDING ? TOMBSTONE
  CAS(&map[pos].hash, pending, HASH_TOMBSTONE);
  // Entry is now invisible to readers (skipped in probe)
}

// Full registration with automatic rollback
bool Register(IdSlot* map, uint32 hash, uint16 id) {
  int pos = RegisterPhase1(map, hash);
  if (pos < 0) return false;

  try {
    // Upstream processing that might throw
    var result = UpstreamProcess(hash, id);
    ValidateResult(result);

    // Success ? commit
    RegisterPhase2_Commit(map, pos, id);
    return true;
  } catch {
    // Failure ? automatic rollback
    RegisterPhase2_Rollback(map, pos);
    return false;
  }
}

// Lookup: PENDING entries are invisible
bool Lookup(IdSlot* map, uint32 hash, uint16* outId) {
  uint32 idx = hash % 8192;
  for (uint32 i = 0; i < 8192; i++) {
    uint32 pos = (idx + i) % 8192;
    uint32 h = AtomicLoad(&map[pos].hash);
    if (h == HASH_EMPTY) return false;
    if (h & PENDING_BIT) continue; // skip pending
    if (h == HASH_TOMBSTONE) continue; // skip tombstone
    if (h == hash) {
      *outId = map[pos].id;
      return true;
    }
  }
  return false;
}

CONSTRAINTS:
  Ģ Zero-Lock: All phase transitions use CAS Ś no mutex or spinlock.
  Ģ Zero-Allocation: PENDING state is encoded in the high bit of the existing hash field Ś zero extra memory.
  Ģ Correctness: PENDING entries are invisible to readers by construction Ś no partial registration can ever be observed.

RATIONALE:
Two-phase commit is a well-established pattern for atomic operations. By encoding the PENDING state in the high bit of the hash field, we avoid any additional memory or allocation. The key insight is that PENDING entries are structurally invisible to readers (the lookup function skips them), which means a failed registration requires no coordination with concurrent readers Ś it simply transitions to TOMBSTONE and is cleaned lazily.

  ADR-003  [CRITICAL] Sideband Cleanup vs Pool Release Ordering
????????????????????????????????????????????????????????????????????????

  VULNERABILITY:

[ADR-001] FSM Class-to-Struct Conversion Ś Torn-Read Isolation via Epoch-Stamped Immutable Snapshots
Severity: CRITICAL | Status: PROPOSED

FLAW:
The primary FSM is a reference type with 12+ public fields including DataBuffers and Pointers. Packing only the State field into an atomic 64-bit long leaves every other field vulnerable to torn reads under concurrent access, violating the assumed linearizability of the FSM.

CONTEXT:
In a zero-allocation, lock-free kernel context, reference-type FSMs are tempting because they allow partial mutation. However, this creates a fundamental split-brain hazard: the atomic state field can advance while companion fields (ring-buffer heads, pointer epochs, data buffer indices) still reflect the prior epoch. Any reader observing the state transition concurrently may act on stale ancillary data.

DECISION:
Decompose the FSM into two distinct layers:

**Layer 1 Ś Atomic Control Plane (64-bit word):** Pack the FSM state enum, a 20-bit epoch counter, and a 6-bit opcode field into a single `Interlocked`-managed `long`. All state transitions are performed exclusively via `CompareExchange` on this single word. This ensures that state + metadata always move atomically.

**Layer 2 Ś Immutable Snapshot Records (pre-allocated ring):** Allocate a fixed-size ring of N "snapshot slots" at startup (zero-alloc on hot path). Each snapshot is a value-type record containing the full ancillary field set (buffer pointers, lengths, flags). On every valid state transition, the FSM writes the new snapshot into the next ring slot (indexed by the epoch field in the control word), then atomically bumps the control word with the new epoch. Readers always resolve: `snapshot = ring[control_word.epoch & mask]`.

RATIONALE:
Physics analogy: the control word is the photon (indivisible, travels at the speed of the atomic bus), while snapshots are the wavefronts it collapses into. A reader either gets the old wavefront or the new one Ś never a superposition of both. The epoch in the control word acts as the "which-path" detector, making torn reads structurally impossible rather than probabilistically avoided.

CONSEQUENCES:
  1. Torn reads on companion fields become structurally impossible Ś a reader either holds a consistent snapshot or retries.
  2. Ring size (N) must be chosen as a power-of-2 large enough to outlive the worst-case reader lag; this is a tuning constant, not a runtime allocation.
  3. Snapshot records must be pure value types (no internal references) to honour Zero-Allocation; any pointer stored is an index into a separately pre-allocated buffer.
  4. Old snapshots are naturally overwritten after N epochs with no GC involvement.

SOVEREIGN DNA COMPLIANCE:
  Zero-Lock  : ? CompareExchange on one word only; no Mutex, Monitor, or SpinLock.
  Zero-Alloc : ? Ring allocated at startup; hot-path writes only to pre-existing slots.
  Correctness: ? Epoch monotonicity ensures no reader can observe a future snapshot with a past state.

[ADR-002] Hash Map Pool Exhaustion & Tombstone Deletion Ś Overflow Chaining + Robin Hood Tombstones
Severity: CRITICAL | Status: PROPOSED

FLAW:
Flaw A: The pre-allocated ID map (8,192 slots) has no fallback when full Ś exhaustion silently orphans incoming packets. Flaw B: Zeroing a slot on deletion breaks the invariant of linear probing: any probe chain that passed through the deleted slot is now severed, causing false "not found" results for entries beyond that slot.

CONTEXT:
A fixed-capacity open-addressed hash map with linear probing is the correct zero-alloc data structure here. However, two structural invariants must hold simultaneously: (1) the probe chain must be unbroken for all live entries, and (2) the map must degrade gracefully rather than fatally on full capacity. Both flaws stem from treating the map as a simple array of optional values rather than a probe-chain-aware structure.

DECISION:
**Flaw A Resolution Ś Overflow Epoch Redirect:** Reserve a contiguous "overflow band" of M slots (e.g., 256) at the end of the pre-allocated array. These slots are marked with an `OVERFLOW` sentinel type, not a separate allocation. When the primary 8,192-slot region is exhausted, new registrations target the overflow band using a secondary hash (double-hashing). Overflow-band entries carry a TTL epoch counter: if the epoch counter expires before the primary map drains below a threshold, the entry is evicted with a structured error code (OVERFLOW_EVICTED) rather than silently orphaned. This converts silent data loss into a diagnosable, deterministic event.

**Flaw B Resolution Ś Generation-Stamped Tombstones:** On deletion, write a `TOMBSTONE` sentinel into the slot rather than zeroing it. Tombstones preserve the probe chain for all subsequent lookups (the probe continues past tombstones). Tombstones can be reused by insertions (they count as "empty" for write purposes). To prevent tombstone accumulation degrading O(1) to O(N), maintain an atomic tombstone counter. When `tombstone_count > capacity / 4`, trigger an in-place "rehash compact" during a guaranteed idle window (not the hot path): sweep the array sequentially, re-inserting live entries into earlier tombstone slots.

RATIONALE:
Tombstone deletion is the textbook solution to linear-probing chain preservation (Knuth, TAOCP Vol. 3, ¦6.4). The overflow band is analogous to a capacitor in an electrical circuit: it absorbs transient current spikes without blowing the fuse, while the TTL prevents indefinite accumulation. The compact operation is analogous to a scheduled garbage-collect in a stop-the-world phase Ś but scoped only to a bounded array, with no heap involvement.

CONSEQUENCES:
  1. Overflow band size M is a tuning constant; setting M too small increases OVERFLOW_EVICTED frequency under burst load.
  2. Tombstone count must be tracked atomically; the compact operation must be gated by a single CAS to prevent concurrent compaction.
  3. Probe chain correctness is restored: lookups skip tombstones but do not terminate at them, ensuring all live entries beyond a deleted slot remain reachable.
  4. Double-hashing for the overflow band must use a hash function whose stride is coprime to the array size (use prime stride or power-of-2 with odd-stride guarantee).

SOVEREIGN DNA COMPLIANCE:
  Zero-Lock  : ? Tombstone writes and overflow redirects use atomic slot-state CAS; compaction is a single-threaded idle-phase sweep gated by CAS.
  Zero-Alloc : ? Overflow band is contiguous within the original pre-allocated array; no new heap objects created on any path.
  Correctness: ? Probe chain invariant is structurally preserved; exhaustion produces a typed error, not silent data loss.

[ADR-003] Sideband Cleanup vs Pool Release Ordering Ś Dual-Phase Commit with Poison-Slot Backstop
Severity: CRITICAL | Status: PROPOSED

FLAW:
The existing contract releases the pool slot in a `finally` block. If the proposed sideband-first clearing path throws an exception, the `finally` runs and the pool slot is released Ś but the sideband may be in a partially-cleared, corrupted state. Any subsequent allocation of that slot reads poisoned sideband data.

CONTEXT:
The fundamental ordering constraint is: sideband must be verifiably clean before the slot is visible to new allocators. A `finally` block cannot enforce ordering Ś it executes regardless of whether the sideband clear succeeded. This is a structural flaw in using exception-driven control flow for atomic multi-step cleanup.

DECISION:
Replace the single `finally` block with a **Dual-Phase Commit** pattern using a slot-local `CleanupPhase` atomic field:

**Phase 0 Ś Claim for Cleanup:** The releasing thread atomically CAS the slot state from `LIVE` ? `DRAINING`. Any concurrent allocator that sees `DRAINING` skips that slot. The slot is logically withdrawn from the pool but not yet free.

**Phase 1 Ś Sideband Clear:** Execute the sideband clear. If it throws, the slot remains in `DRAINING` Ś it is NOT returned to the pool. A background "poison sweeper" (pre-allocated, single-purpose idle thread or periodic scan) detects slots that have been in `DRAINING` beyond a deadline epoch and re-attempts the sideband clear from a safe, exception-isolated context.

**Phase 2 Ś Release:** Only if Phase 1 completes without exception does the releasing thread CAS the state `DRAINING` ? `FREE`, making the slot available to allocators.

**Backstop:** The poison sweeper holds its own pre-allocated retry buffer. It never allocates. It simply re-runs the idempotent clear and, on success, advances the state to `FREE`. On repeated failure (> N retries), the slot is permanently marked `POISON` and excluded from the pool Ś the pool shrinks by 1 rather than corrupting future allocations.

RATIONALE:
The DRAINING state is a structural firewall: it makes it physically impossible for an allocator to observe a slot that has not completed its cleanup lifecycle. This is equivalent to a filesystem journal commit point Ś you do not mark a block as free until the journal confirms the write. The poison sweeper is the "journal recovery" path: it handles the exceptional case without corrupting the main path.

CONSEQUENCES:
  1. Pool capacity can transiently shrink by the number of slots in DRAINING or POISON state; sizing must account for this headroom.
  2. The sideband clear operation must be idempotent (safe to call multiple times) for the poison sweeper to retry safely.
  3. DRAINING detection in the allocator requires only a state enum read Ś no lock, no allocation.
  4. POISON slots emit a structured metric event; sustained POISON accumulation indicates a systemic sideband failure requiring operator attention.

SOVEREIGN DNA COMPLIANCE:
  Zero-Lock  : ? All phase transitions use single CAS on the slot state field; no locks anywhere in the path.
  Zero-Alloc : ? Poison sweeper uses a pre-allocated retry buffer; DRAINING/POISON states are enum values in the existing slot struct.
  Correctness: ? The DRAINING firewall makes allocator-observing-dirty-slot impossible by construction.

[ADR-004] ABA Risk & Slot Generation Counters Ś Bit-Multiplexed Generation Tags in Atomic Words
Severity: CRITICAL | Status: PROPOSED

FLAW:
Flaw A: The Circuit Breaker state lacks a generation counter, so a thread that observes state S0, gets preempted, watches the system cycle S0?S1?S2?S0, and then performs a CAS will succeed incorrectly Ś a textbook ABA race. Flaw B: The Dispatch Slot lacks a generation field, meaning a delayed callback arriving after slot reallocation reads the new slot owner's sideband Ś silent data cross-contamination.

CONTEXT:
ABA is the fundamental hazard of compare-exchange on values that can be recycled. The classic solution Ś tagged pointers Ś requires pointer-sized tags. In a zero-allocation kernel model, every state-carrying word must embed its own generation tag without requiring additional heap objects or pointer indirection.

DECISION:
**Resolution for Both Flaws Ś Bit-Multiplexed Generation Tags:**

For the **Circuit Breaker**: The existing state enum requires at most 8 distinct values (3 bits). Pack it into the low 3 bits of the atomic `long`. Reserve the upper 32 bits as a monotonically-increasing generation counter. The remaining middle bits carry metadata (trip counter, half-open probe count). All transitions use `CompareExchange` on the full 64-bit word Ś an ABA race requires the generation to wrap around 2│▓ times while a thread is preempted, which is physically implausible.

For the **Dispatch Slot**: Embed a 16-bit generation counter directly inside the `DispatchSlot` value-type struct. When a slot is allocated, the counter is incremented atomically. Every callback captures the generation at the moment of registration as part of its closure (an index, not a reference). On callback invocation, the captured generation is compared to the slot's current generation via a simple read Ś mismatch means the slot was reallocated and the callback discards itself. No lock. No allocation.

RATIONALE:
This follows the "lock-free tagged pointer" idiom but without actual pointers Ś we are tagging integer indices, which costs nothing. The generation counter transforms the flat CAS space from "did the value match?" into "did this exact lifecycle match?", converting a probabilistic hazard into a structural impossibility within any physically observable time window.

CONSEQUENCES:
  1. Generation counters must be initialized to distinct non-zero values at startup to avoid false matches against zeroed memory.
  2. For Circuit Breaker, the 32-bit generation counter wraps at 2│▓ Ś ensure the system cannot reach this in a single uptime window (at 1 GHz transition rate, wrap occurs after ~4 seconds; use 48-bit if required).
  3. For Dispatch Slots, the 16-bit generation counter wraps at 65,536 Ś acceptable if slot reuse rate is bounded; use 32-bit for high-throughput paths.
  4. Callbacks must be written as generation-checking wrappers, not raw delegates Ś this is a structural API constraint enforceable at compile time.

SOVEREIGN DNA COMPLIANCE:
  Zero-Lock  : ? All generation checks are a read + equality comparison; all transitions are CAS on the packed word.
  Zero-Alloc : ? Generation field is embedded in the existing value-type struct; no additional objects created.
  Correctness: ? ABA requires generation wraparound within preemption window Ś physically implausible for correctly-sized counters.

[ADR-005] Registration Rollback Ś Scope-Guarded Atomic Claim with Structural Commit/Abort
Severity: CRITICAL | Status: PROPOSED

FLAW:
If a worker registers a packet in the ID map and the upstream processing subsequently throws, the mapping remains in the map permanently Ś the slot is never released, and the ID is leaked. There is no rollback mechanism, causing the map to fill with phantom entries over time.

CONTEXT:
The root cause is that "registration" and "processing" are treated as a single, non-transactional step. When the processing phase fails, there is no compensating action because the registration has no lifecycle awareness Ś it does not know whether the work it sponsored succeeded or failed.

DECISION:
Introduce a **Scope-Guarded Registration** pattern using a value-type `RegistrationScope` struct and a two-state slot flag:

**Slot States:** `TENTATIVE` and `COMMITTED`. A freshly registered slot is `TENTATIVE`. Only the explicit `.Commit()` call advances it to `COMMITTED`. `COMMITTED` slots are permanent (until explicit delete). `TENTATIVE` slots are automatically rolled back.

**Structural Rollback:** The `RegistrationScope` struct implements `IDisposable`. If `.Dispose()` is called without a preceding `.Commit()`, it atomically clears the map slot (CAS `TENTATIVE` ? `EMPTY` with tombstone if needed). The caller uses a `using` block Ś the compiler enforces that `Dispose` is always called, making the rollback structural, not optional.

**Two-Phase Lookup:** Readers that encounter a `TENTATIVE` slot treat it as absent Ś the slot is invisible to consumers until committed. This prevents a reader from acting on a registration that may yet be rolled back.

**Zero-Allocation Guarantee:** `RegistrationScope` is a value type (struct). It carries only the slot index and captured generation as fields. No delegate, no closure, no heap object is created.

RATIONALE:
This mirrors the RAII (Resource Acquisition Is Initialization) pattern from systems programming, applied to map slot lifecycle. The `using` block is the structural guarantee: the compiler transforms it into a try/finally, ensuring Dispose runs even on exception Ś but here the Dispose IS the rollback, not a release of an already-committed resource. The key insight is that the resource (slot) is not fully "acquired" until `.Commit()` is called.

CONSEQUENCES:
  1. `TENTATIVE` slots are invisible to readers Ś this is correct but means a producer cannot read back its own registration until it commits.
  2. The `using` scope must span the entire processing window (registration ? processing ? commit); scoping it too narrowly risks premature rollback.
  3. If the thread is killed (process abort), TENTATIVE slots persist until the startup/recovery scan Ś include a startup sweep that clears all TENTATIVE entries.
  4. Two-phase lookup adds one state-check to every map read Ś this is a single branch on a byte field, effectively free.

SOVEREIGN DNA COMPLIANCE:
  Zero-Lock  : ? Commit and rollback both use CAS on the slot state byte; no locks.
  Zero-Alloc : ? RegistrationScope is a value-type struct; compiler-managed stack allocation only.
  Correctness: ? Compiler enforces Dispose via using; TENTATIVE invisibility prevents consumers from acting on uncommitted state.

ADR-001: FSM Class-to-Struct Thread-Safe Reference Field Model
Severity: Critical | Status: Resolved
Constraints: Zero-Lock, Zero-Alloc, Correctness by Construction

CONTEXT:
The primary Finite State Machine (FSM) is currently implemented as a reference type (class) with 12+ public fields, including reference-typed fields (DataBuffers, Pointers, delegates). The proposed optimization packs only the State enum into an atomic 64-bit long. This leaves the reference fields unprotected from torn reads: a concurrent thread can observe a partially updated DataBuffer pointer while the State CAS has already committed, creating a consistency violation where the FSM reports a state that does not match its data.

DECISION:
Implement a Lock-Free Double-Buffer Reference Swapping protocol using an Immutable Ring Buffer paired with a Generation-Gated Access pattern.

1. Immutable Sideband Ring Buffer: Pre-allocate a fixed-size ring buffer of "DataBundle" structs at initialization. Each bundle is immutable once written and contains all reference fields (DataBuffer pointers, callback delegates, metadata). Being immutable, any thread that reads a bundle sees a consistent snapshot Ś no field can be torn because the bundle is never mutated in-place.

2. Atomic Bundle Pointer Swap: The FSM holds a single 64-bit atomic field pointing to the current ring buffer slot index. When the FSM transitions state and needs to update reference data, it writes the new DataBundle into the next available ring buffer slot (pre-allocated, zero runtime alloc), then atomically CAS-es the bundle pointer. The old bundle remains valid until all in-flight readers have drained.

3. Generation-Gated Read Protocol: The 64-bit atomic state word is split into two fields: 48 bits for FSM state + metadata, and 16 bits for a monotonically increasing generation counter (0¢65535). Every time the bundle pointer is swapped, the generation counter increments. Readers read the generation counter first, then the bundle pointer, then the state, then re-read the generation. If the generation changed between reads, the snapshot is stale and the reader retries. This provides a lock-free consistent read of the entire FSM surface without any allocation.

CONSEQUENCES:
  Ģ Positive: Eliminates torn reads entirely Ś reference fields are always observed as a consistent snapshot aligned with the FSM state.
  Ģ Positive: Zero runtime allocation Ś the ring buffer is pre-allocated at initialization. Only index arithmetic occurs on the hot path.
  Ģ Positive: No locks Ś all synchronization is via atomic CAS and generation-counter retry loops.
  Ģ Trade-off: The ring buffer has a fixed capacity. If transitions occur faster than readers drain, the buffer can wrap and overwrite a bundle still being read. Mitigation: size the ring buffer to exceed the maximum possible in-flight reader count ū 2 (double buffering), making overwrite impossible by construction.
  Ģ Trade-off: Generation counter wraps after 65536 transitions. With a 16-bit counter, this requires 65K state transitions between a single reader's first and second generation read Ś physically impossible on any real system. Even if it occurred, the reader would detect a mismatch and retry, resulting in a harmless spurious retry.

ADR-002: Zero-Alloc Overflow Chain & Tombstone Deletion Strategy
Severity: Critical | Status: Resolved
Constraints: Zero-Alloc, Correctness by Construction

CONTEXT:
The pre-allocated hash map pool has 8192 fixed slots. Two flaws exist:
(A) Pool Exhaustion: When all 8192 slots are occupied, new packet registrations fail silently, causing packets to become orphaned with no routing entry.
(B) Linear Probing Deletion: The current deletion strategy simply zeroes out the slot entry (sets key to 0). This breaks linear probing chains: a subsequent lookup for a key that was inserted after the deleted key will terminate early at the zeroed slot, never finding the target key.

DECISION:
Implement a Zero-Allocation Overflow Chain using embedded packet header fields, combined with a Tombstone Deletion Strategy for linear probing integrity.

Pool Exhaustion Ś Embedded Overflow Chain:
Repurpose an existing unused 8-byte field in each packet header (e.g., the lower 64 bits of a nanosecond timestamp, or a reserved alignment padding field) as a "next overflow index" pointer. When the primary 8192-slot pool is exhausted, new registrations are chained into an overflow linked list rooted at a sentinel entry in the main table. The overflow chain uses indices into the existing packet array Ś no new memory is allocated. Each overflow entry's "next" field points to the next overflow packet. Lookup follows the chain: check primary slot, if tombstone/exhausted, follow overflow chain.

Deletion Ś Tombstone Pattern:
Instead of zeroing a deleted entry, mark it with a reserved tombstone sentinel value (e.g., key = 0xFFFFFFFF_FFFFFFFE, which is an invalid packet ID by construction). Linear probing treats tombstones as "continue searching" during lookups, but as "available for insertion" during writes. This preserves probe chain integrity.

Periodic Off-Hot-Path Compaction: A background worker (not on the hot path) periodically compacts tombstones by re-inserting all live entries into a fresh region of the table, then atomically swapping the table pointer. This reclaims tombstone slots without blocking concurrent operations.

CONSEQUENCES:
  Ģ Positive: Pool exhaustion no longer causes orphaned packets Ś overflow chain provides unbounded (but bounded by physical memory) capacity.
  Ģ Positive: Zero allocation Ś overflow chain reuses existing packet header fields. Tombstone sentinel is a constant value, no alloc.
  Ģ Positive: Linear probing integrity is preserved Ś tombstones maintain probe chain continuity.
  Ģ Positive: Compaction runs off the hot path, ensuring no GC pressure or latency impact on packet processing.
  Ģ Trade-off: Overflow chain lookups are O(n) in chain length. Mitigation: the overflow chain is only traversed when the primary pool is full (a rare condition), and compaction periodically drains it.
  Ģ Trade-off: Tombstones consume slots until compaction. If deletions are extremely frequent with no insertions, the table can fill with tombstones. Mitigation: compaction threshold triggers when tombstone density exceeds 30%.
  Ģ Correctness by Construction: The tombstone sentinel value (0xFFFFFFFF_FFFFFFFE) is chosen to be an invalid packet ID by the upstream protocol Ś it can never collide with a legitimate key.

IMPLEMENTATION:

ADR-003: Fail-Safe Structural Ordering for Sideband & Pool Cleanup
Severity: Critical | Status: Resolved
Constraints: Zero-Lock, Correctness by Construction

CONTEXT:
The existing cleanup contract releases the pool slot in a `finally` block. The proposed modification performs sideband clearing *before* pool release to ensure no stale references remain. However, if the sideband clearing operation throws an exception, the `finally` block still releases the pool slot Ś but the sideband is left in a partially cleared state. Conversely, if sideband clearing is moved *outside* the `finally` and throws, the pool slot is never released, causing a permanent slot leak.

DECISION:
Implement a Two-Phase State-Machine Cleanup Protocol where the slot's lifecycle state machine structurally prevents invalid ordering.

Phase 1 Ś Mark Pending Cleanup: Atomically transition the slot state from Active ? PendingCleanup via CAS. This transition is idempotent and cannot fail (it only checks that the current state is Active; if it's already PendingCleanup or Released, the operation is a no-op).

Phase 2 Ś Execute Sideband Cleanup: Perform the sideband clearing operation. If this throws, the slot remains in PendingCleanup state Ś NOT Released. A background compaction sweeper detects slots stuck in PendingCleanup and retries the sideband cleanup.

Phase 3 Ś Release Pool Slot: Only after sideband cleanup succeeds does the slot transition to Released. The pool slot is then reclaimed.

The state machine is a sealed struct with a private constructor and only valid transitions exposed as methods. Invalid transitions throw at compile time (type-level prevention) or return false at runtime (state check). The "PendingCleanup" state acts as a circuit breaker: even if sideband cleanup fails, the slot is never released, and the sweeper guarantees eventual consistency.

CONSEQUENCES:
  Ģ Positive: Zero slot leaks Ś the slot is only released after successful sideband cleanup, and the sweeper guarantees eventual cleanup even on persistent failures.
  Ģ Positive: No locks Ś all state transitions are atomic CAS operations.
  Ģ Positive: Correctness by Construction Ś the sealed state machine struct only exposes valid transitions, making invalid ordering impossible to express in code.
  Ģ Positive: Idempotent Ś retrying sideband cleanup is safe because it operates on the PendingCleanup state and checks current sideband contents before clearing.
  Ģ Trade-off: Requires a background sweeper thread (or timer-based callback) to detect and retry stuck PendingCleanup slots. This is a minor resource cost.
  Ģ Trade-off: Slots in PendingCleanup state are temporarily unavailable for reuse. In the worst case (sideband cleanup consistently fails), slots accumulate until the sweeper resolves them.

IMPLEMENTATION:

  // State Machine Slot (Correctness by Construction)
// Sealed state machine Ś invalid transitions impossible.
// The state is an 8-bit atomic value with defined transitions.

public readonly struct CleanupState {
    private readonly byte _raw;

ADR-004: Generation Counter Embedding for ABA Prevention
Severity: Critical | Status: Resolved
Constraints: Zero-Lock, Zero-Alloc, Correctness by Construction

CONTEXT:
Two ABA vulnerabilities exist in the current architecture:
(A) Circuit Breaker: The circuit breaker state is a simple enum (Closed/Open/HalfOpen) stored in an atomic field. Without a generation counter, the ABA problem applies: Thread 1 reads state=Closed, Thread 2 transitions Closed?Open?Closed, Thread 1's CAS succeeds because it still sees Closed Ś but the circuit breaker has experienced a full cycle that Thread 1 is unaware of.
(B) Dispatch Slot: When a dispatch slot is released and reallocated, a delayed callback from the previous allocation can read the new slot's sideband data, causing data cross-contamination. The slot lacks a generation field to distinguish between "my slot" and "a different slot that happens to be at the same index."

DECISION:
Embed monotonically increasing generation counters into both the Circuit Breaker state word and the Dispatch Slot, using bit-packing to maintain zero-allocation.

Circuit Breaker Ś Generation-Tagged State Word:
The circuit breaker's 64-bit atomic word is repartitioned: 48 bits for state + metadata, 16 bits for a generation counter. Every state transition (Closed?Open, Open?HalfOpen, HalfOpen?Closed) increments the generation. A CAS operation that includes the generation in its expected value will fail if any intermediate transition occurred, preventing the ABA race. The 16-bit counter (65536 values) is sufficient because the circuit breaker cannot cycle 65K times between a single thread's read and CAS Ś this would require nanosecond-scale full cycles sustained over milliseconds, which is physically impossible given the minimum circuit breaker timeout thresholds.

Dispatch Slot Ś Slot-Local Generation Field:
Each dispatch slot in the pre-allocated pool includes a 32-bit generation field (part of the slot struct, zero additional allocation). When a slot is released, its generation is incremented before the slot is made available for reallocation. Callbacks carry a copy of the generation from when they were registered. Before accessing slot data, the callback verifies: if (slot.Generation == callback.Generation) Ś access is safe. If not, the slot has been recycled and the callback must abort.

Both mechanisms are zero-allocation (generation counters are embedded in existing structures), lock-free (atomic CAS on the full word), and provide correctness by construction (the generation is part of the slot/state identity Ś mismatched generation means "not my data").

CONSEQUENCES:
  Ģ Positive: ABA race eliminated for both circuit breaker and dispatch slots Ś any intermediate state change invalidates stale CAS operations.
  Ģ Positive: Zero allocation Ś generation counters are bit-packed into existing atomic words or slot structs.
  Ģ Positive: No locks Ś all generation checks and updates are atomic operations.
  Ģ Positive: Data cross-contamination prevented Ś callbacks with stale generation cannot access recycled slot data.
  Ģ Trade-off: Circuit breaker generation wraps after 65536 transitions. Mitigation: the minimum time between circuit breaker cycles (timeout + recovery period) makes 65K cycles between a single read and CAS physically impossible. Even on wrap, the generation still changes, so ABA is still detected.
  Ģ Trade-off: Dispatch slot generation wraps after 4 billion reallocations (32-bit counter). Mitigation: at realistic slot turnover rates, this would take centuries. If needed, the counter can be widened or the slot can be quarantined on wrap.

IMPLEMENTATION:

ADR-005: Lock-Free Registration Rollback via Per-Thread Undo Log
Severity: Significant | Status: Resolved
Constraints: Zero-Lock, Zero-Alloc

CONTEXT:
When a worker registers a packet in the hash map, it creates a mapping from packet ID to dispatch slot index. If the upstream processing (which occurs after registration) throws an exception, the mapping leaks permanently Ś it remains in the hash map pointing to a slot that will never be used, and the packet ID is permanently consumed. The current design has no rollback mechanism.

DECISION:
Implement a Per-Thread Circular Undo Log for registration rollback, using a pre-allocated fixed-size buffer per worker thread.

Each worker thread owns a pre-allocated circular buffer (undo log) with a fixed capacity (e.g., 256 entries). Before registering a packet in the hash map, the worker writes an undo entry to its local log: (packetId, slotIndex, hash, timestamp). The undo entry is written with a status of "pending."

If upstream processing succeeds, the worker atomically marks the undo entry as "committed." If upstream processing fails, the worker scans its undo log from the tail, finds the most recent "pending" entry, and atomically removes the hash map mapping using CAS with a generation check (from ADR-004). The undo entry is then marked "rolled back."

The undo log is circular: once full, the oldest entries are overwritten. This is safe because: (1) entries are only overwritten after being committed or rolled back, (2) if the log fills with pending entries, it means the worker has 256 consecutive failed registrations without cleanup Ś at which point the worker self-terminates (a safety valve). The log is never allocated at runtime Ś it is part of the worker's TLS (thread-local storage) block.

CONSEQUENCES:
  Ģ Positive: Zero permanent registration leaks Ś every failed registration is rolled back.
  Ģ Positive: Zero allocation Ś undo log is pre-allocated as part of worker TLS.
  Ģ Positive: No locks Ś rollback uses CAS operations on the hash map entries.
  Ģ Positive: Thread-local Ś each worker's undo log is independent, no cross-thread contention.
  Ģ Trade-off: Undo log has finite capacity (256 entries per worker). If a worker accumulates 256 pending registrations without completing any, it self-terminates. This is a deliberate safety valve Ś it indicates a systemic failure that should not be silently absorbed.
  Ģ Trade-off: Rollback requires a CAS on the hash map entry. In the extremely rare case that the CAS fails (because another thread already modified the entry), the worker retries with exponential backoff. The generation counter from ADR-004 ensures the CAS target is unambiguous.

IMPLEMENTATION:

        ref var entry = ref _entries[idx];

        entry.PacketId   = packetId;
        entry.SlotIndex  = slotIndex;
        entry.Hash       = hash;
        entry.Timestamp  = Stopwatch.GetTimestamp();
        entry.Status     = (byte)UndoEntryStatus.Pending;

        if (entry.Status != (byte)UndoEntryStatus.Empty)

    public void Commit(int entryIndex) {
        _entries[entryIndex].Status =
            (byte)UndoEntryStatus.Committed;
    }

    // Rollback (upstream failed).
    public void Rollback(ref HashMapSlot[] table) {
        for (int i = 0; i < _entries.Length; i++) {
            int idx = (_writeIndex - 1 - i + _entries.Length)
                      % _entries.Length;

                continue;

            Interlocked.CompareExchange(
                ref table[slotIdx].Key,
                TOMBSTONE, // replace with tombstone (ADR-002)
                entry.PacketId);

            entry.Status = (byte)UndoEntryStatus.RolledBack;
            return; // rolled back most recent pending entry
        }
    }
}

void RegisterAndProcess(
    UndoLog       undoLog,
    HashMapSlot[]  hashMap,
    ulong         packetId,
    int           hash,

    catch {

ADR-001: Frozen Sidecar Tuple via Atomic Pointer Swap
Severity: CRITICAL | Vulnerability: FSM Class-to-Struct Conversion Mismatch
????????????????????????????????????????????????????????????????????????????????

VULNERABILITY ANALYSIS:
The primary FSM is a reference type with 12+ public fields (DataBuffers, Pointers, State, etc.). Packing only the State field into an atomic 64-bit long provides atomicity for that single field Ś but leaves every other reference field vulnerable to torn reads under concurrent mutation. A thread reading the FSM may observe a partially-updated object where State is new but DataBuffers still points to the previous cycle's buffer.

PROPOSED MECHANISM:
Principle: Collapse all mutable reference fields into a single immutable "Sidecar" struct. The FSM holds exactly one atomic pointer to the current Sidecar instance. Mutations construct a new Sidecar and CAS the pointer Ś readers always see a consistent snapshot.

Construction:
1. Define a frozen struct Sidecar containing all 12 reference fields (buffers, pointers, config refs, etc.). This struct is immutable once published.
2. The FSM holds a single AtomicReference<Sidecar> Ś the *only* mutable state besides the packed state word.
3. Pre-allocate a fixed pool of Sidecar instances (e.g., 2 ū worker-count) at system initialization. This pool is never resized.
4. On state transition:
   a. Claim a fresh Sidecar from the pool (via atomic index bump, modulo pool size).
   b. Copy current Sidecar fields into the new instance, applying the mutation.
   c. CAS the AtomicReference to the new Sidecar.
   d. Return the old Sidecar to the pool.
5. Readers call sidecarRef.get() once, then read all fields from that snapshot Ś guaranteeing consistency without locks.

ADR-002: Tombstone Deletion & L1 Overflow Ring Buffer
Severity: CRITICAL | Vulnerability: Hash Map Pool Exhaustion & Deletion
????????????????????????????????????????????????????????????????????????????????

VULNERABILITY ANALYSIS:
Flaw A: The pre-allocated ID map has 8192 slots. When all slots are occupied, new registrations fail silently Ś packets become orphaned with no ID mapping, leading to untraceable packet loss in production.

Flaw B: Linear probing relies on contiguous chains from the hash index. When an entry is zeroed on removal, it creates a "hole" in the probe chain. Subsequent lookups for entries that were probed past this hole will terminate early (seeing zero = empty), returning false negatives. The entry still exists but is permanently unreachable.

PROPOSED MECHANISM:
Deletion Ś Tombstone Sentinel:
Replace the zero-on-delete strategy with a TOMBSTONE sentinel value (e.g., 0xDEAD_DEAD_DEAD_DEAD). Linear probing treats tombstones as:
- Occupied for chain continuation: probing continues past a tombstone.
- Empty for insertion: a tombstone slot may be reused by new registrations.

This preserves all probe chains while allowing slot reuse. Periodic compaction (during known-quiescent periods) can batch-clear tombstones and re-probe surviving entries.

Exhaustion Ś L1 Overflow Ring Buffer:
Maintain a secondary pre-allocated ring buffer (256 slots) as a spillover zone:
1. Primary map lookup: standard linear probe.
2. On primary-map-full: insert into the ring buffer at atomic_head.fetch_add(1) % RING_SIZE.
3. Lookup order: primary ? ring buffer scan.
4. The ring buffer uses atomic stamps (slot_id + generation) for safe concurrent access.
5. If the ring buffer also saturates: apply explicit back-pressure (return Exhausted error to caller). Never silently drop.

Sizing guarantee: 8192 + 256 = 8448 concurrent registrations. The ring buffer's small size makes linear scan acceptable (~256 comparisons = ~2?s at 128ns/compare on modern x86).

REFERENCE IMPLEMENTATION:
const TOMBSTONE: u64 = 0xDEAD_DEAD_DEAD_DEAD;
const EMPTY: u64     = 0x0000_0000_0000_0000;

struct Slot {
    key: AtomicU64,
    value: AtomicU64,
}

fn delete(map: &[Slot], hash: u64, key: u64) -> bool {
    let mut idx = hash as usize % map.len();
    loop {
        let k = map[idx].key.load(Acquire);
        if k == key {
            map[idx].key.store(TOMBSTONE, Release);
            // value left for tombstone reuse or compaction
            return true;
        }
        if k == EMPTY { return false; } // end of chain
        idx = (idx + 1) % map.len();
    }
}

fn insert(map: &[Slot], ring: &RingBuffer, key: u64, val: u64)
    -> Result<(), Exhausted> 
{
    let hash = fxhash(key);
    let mut idx = hash as usize % map.len();
    let mut first_tomb = None;
    
    loop {
        let k = map[idx].key.load(Acquire);
        if k == EMPTY || k == key {
            // insert here (CAS for thread safety)
            return cas_insert(&map[idx], k, key, val);
        }
        if k == TOMBSTONE && first_tomb.is_none() {
            first_tomb = Some(idx);
        }
        idx = (idx + 1) % map.len();
        if idx == hash as usize % map.len() {
            // Full probe Ś try tombstone reuse, then overflow
            if let Some(t) = first_tomb {
                return cas_insert(&map[t], TOMBSTONE, key, val);
            }
            return ring.push(key, val); // L1 overflow
        }
    }
}

  Zero-Lock: All map operations use AtomicU64 CAS. The ring buffer uses atomic head/tail indices. No mutexes or locks anywhere in the path.

  Correctness by Construction: Tombstones structurally prevent chain breakage Ś it is impossible for a deletion to make a live entry unreachable. The ring buffer's atomic stamps prevent ABA on slot reuse (see ADR-004).

RATIONALE:
Tombstone deletion is the standard approach in lock-free hash maps (used by Java's ConcurrentHashMap, F14, Swiss Tables). The L1 ring buffer trades O(1) lookup for O(n) on overflow Ś but the overflow is sized so that O(256) is fast enough to be negligible. Explicit back-pressure on double-overflow is architecturally superior to silent loss because it makes the failure observable and recoverable upstream.

ADR-003: Outer-Pool / Inner-Sideband Nested Guard Ordering
Severity: SIGNIFICANT | Vulnerability: Sideband Cleanup vs Pool Release Ordering
????????????????????????????????????????????????????????????????????????????????

VULNERABILITY ANALYSIS:
The current contract releases the pool slot in a finally block. The proposed "sideband-first" clearing could throw an exception, which would bypass the pool release and permanently leak the slot. The ordering is:


try {
    clear_sideband(slot)    // ? can throw
} finally {
    release_pool(slot)      // ? skipped if above throws... wait, no:
}                           //     finally DOES run, but if sideband
                            //     throws AGAIN during cleanup...


Actually, in most runtimes, the finally block WILL execute even if the try block throws. The real flaw is subtler: if clear_sideband throws, the exception propagates through finally, and if release_pool inside finally also fails (e.g., due to corrupted state from the failed sideband cleanup), the slot leaks. Additionally, if the sideband cleanup is reordered to run AFTER pool release (the other direction), then a thread could acquire the slot before sideband data is cleared Ś data cross-contamination.

PROPOSED MECHANISM:
Structural Rule: Outer-Pool, Inner-Sideband

The pool release is the outermost guard. The sideband cleanup is nested inside, with its own catch that absorbs all failures:


slot = acquire_pool()
try {                                    // OUTER: pool guard
    try {                                // INNER: sideband cleanup
        clear_sideband(slot)
    } catch (e) {
        log_atomic(e)                    // swallow Ś never propagate
    }
} finally {
    release_pool(slot)                   // ALWAYS runs, ALWAYS succeeds
}


Key invariants:
1. release_pool() is in the outermost finally Ś it runs regardless of sideband success/failure.
2. release_pool() itself must be infallible (it only resets an atomic index Ś it cannot throw).
3. Sideband failures are caught and logged atomically (e.g., write to a pre-allocated error ring buffer) but never propagate.
4. The sideband cleanup runs BEFORE pool release Ś so the slot's sideband is always clean when the next consumer acquires it.

Dual-release safety: If the runtime supports it, use a scope_guard / defer pattern where the pool release is registered at acquisition time and executes at scope exit regardless of control flow Ś making it structurally impossible to forget.

REFERENCE IMPLEMENTATION:

This is the inverse of the RAII pattern used in C++ (where destructors run in reverse order of construction). By making the pool release the outermost guard, we guarantee it runs last. ADR-004: Packed Generation Counters in State & Slot Words
Severity: CRITICAL | Vulnerability: ABA Risk & Slot Generation Matching
????????????????????????????????????????????????????????????????????????????????

VULNERABILITY ANALYSIS:
Flaw A (Circuit Breaker): The Circuit Breaker's state word lacks a generation counter. Under the ABA race, a thread reads state=A, gets descheduled, other threads transition A?B?A, and the original thread's CAS succeeds because the value matches Ś but the semantic meaning has changed. The thread acts on stale assumptions about the circuit's history.

Flaw B (Dispatch Slot): The Dispatch Slot has no generation field. When a slot is released and reallocated, a delayed callback from the previous occupant can read the new occupant's sideband data Ś causing data cross-contamination between unrelated operations.

PROPOSED MECHANISM:
Bit Layout of the 64-bit State Word:


?????????????????????????????????????????????????????????????????????
?  Epoch (32 bits)                ? Generation (16b) ? State (16b)  ?
?  Bits [63:32]                   ? Bits [31:16]     ? Bits [15:0]  ?
?????????????????????????????????????????????????????????????????????


- State (16 bits): FSM state enum (up to 65,536 states Ś far more than needed).
- Generation (16 bits): Incremented on every state transition. Wraps at 65,535. CAS must match both state AND generation to succeed. ABA is prevented because the generation will have changed even if the state cycles back.
- Epoch (32 bits): Global monotonic counter incremented on slot acquisition. Used for dispatch slot identity. A delayed callback compares its captured epoch against the slot's current epoch Ś mismatch means the slot has been recycled and the callback is discarded.

Circuit Breaker: Every transition increments generation. CAS failure means another thread transitioned first Ś retry with updated word.

Dispatch Slot: acquire() copies the current epoch into the returned handle. release() increments epoch and clears sideband. Any callback carrying a stale epoch is structurally rejected.

Space cost: Zero. These bits are carved from the existing 64-bit atomic word Ś no additional memory.

// Bit manipulation constants
const STATE_MASK:  u64 = 0x0000_0000_0000_FFFF;
const GEN_MASK:    u64 = 0x0000_0000_FFFF_0000;
const EPOCH_MASK:  u64 = 0xFFFF_FFFF_0000_0000;
const GEN_SHIFT:   u32 = 16;
const EPOCH_SHIFT: u32 = 32;

#[inline]
fn pack(state: u16, gen: u16, epoch: u32) -> u64 {
    (epoch as u64) << EPOCH_SHIFT
    | (gen as u64) << GEN_SHIFT
    | (state as u64)
}

#[inline]
fn unpack(word: u64) -> (u16, u16, u32) {
    let state = (word & STATE_MASK) as u16;
    let gen   = ((word & GEN_MASK) >> GEN_SHIFT) as u16;
    let epoch = ((word & EPOCH_MASK) >> EPOCH_SHIFT) as u32;
    (state, gen, epoch)
}

// Circuit Breaker transition Ś ABA-safe
fn cb_transition(cb: &AtomicU64, from: u16, to: u16) -> Result<(), CASFailed> {
    loop {
        let current = cb.load(Acquire);
        let (st, gen, epoch) = unpack(current);
        if st != from { return Err(StateMismatch); }
        
        let next = pack(to, gen.wrapping_add(1), epoch);
        match cb.compare_exchange(current, next, AcqRel, Acquire) {
            Ok(_) => return Ok(()),
            Err(_) => continue, // retry Ś another thread transitioned
        }
    }
}

// Dispatch slot Ś generation-validated callback
fn dispatch_callback(slot: &DispatchSlot, handle: DispatchHandle) {
    let current = slot.word.load(Acquire);
    let (_, _, current_epoch) = unpack(current);
    
    if handle.epoch != current_epoch {
        // Stale callback Ś slot has been recycled. Discard silently.
        METRICS.stale_callbacks.fetch_add(1, Relaxed);
        return;
    }
    // Safe to proceed Ś this callback belongs to the current occupant
    execute_callback(handle);
}

  Zero-Lock: Pure CAS on a single AtomicU64. No locks, no spin-waits beyond CAS retry loops.

  Correctness by Construction: CAS matches on (state + generation + epoch) Ś it is structurally impossible for a stale operation to succeed because the generation will have advanced. Wrapped generations (after 2^16 transitions) are handled by the epoch field providing additional disambiguation.

RATIONALE:
This is the standard technique for ABA prevention in lock-free literature, used in hazard pointers, epoch-based reclamation, and seqlocks. The 16-bit generation provides 65,535 transitions before wrapping, which at 10M transitions/sec wraps in ~6.5ms Ś but the epoch field (32 bits) extends this to 2^32 ū 6.5ms ? 312 days of continuous wrapping before any ambiguity is theoretically possible. In practice, no slot survives that many transitions without being freed.

ADR-005: Two-Phase Atomic Registration with Scoped Rollback
Severity: SIGNIFICANT | Vulnerability: Registration Rollback
????????????????????????????????????????????????????????????????????????????????

VULNERABILITY ANALYSIS:
If a worker registers a packet (writes the ID?Slot mapping) and the upstream processing subsequently throws an exception, the mapping is never cleaned up. The ID remains mapped to a slot that no longer has valid context Ś a permanent leak. Over time, these leaked mappings accumulate and exhaust the ID space (contributing to ADR-002's exhaustion problem).

The flaw is that registration and processing are not in the same transactional scope Ś the mapping is written optimistically with no rollback path.

PROPOSED MECHANISM:
Two-Phase Registration: RESERVE ? COMMIT / ROLLBACK

Define three slot states in the ID map:
- EMPTY = 0x0000_0000_0000_0000 Ś unoccupied
- PENDING = 0x0001_0000_0000_0000 Ś reserved, awaiting commit
- COMMITTED = <packed slot_id + generation> Ś active mapping

Phase 1 Ś RESERVE:
The worker atomically CAS-writes a PENDING sentinel into the ID map slot. This claims the mapping without publishing it.

Phase 2a Ś COMMIT:
If upstream processing succeeds, atomically replace PENDING with the actual packed slot_id + generation.

Phase 2b Ś ROLLBACK:
If upstream processing fails (exception, timeout, validation error), atomically replace PENDING with EMPTY Ś releasing the mapping.

Structural enforcement via scope guard:
The rollback is registered as a scope guard at reservation time. If the scope exits without committing (normal error path OR panic), the guard fires and rolls back. The commit explicitly disarms the guard.

This is the moral equivalent of a database two-phase commit, but using atomic CAS on a single word.

REFERENCE IMPLEMENTATION:
const EMPTY: u64   = 0x0000_0000_0000_0000;
const PENDING: u64 = 0x0001_0000_0000_0000;

struct RegistrationGuard<'a> {
    slot: &'a AtomicU64,
    armed: bool,
}

impl<'a> RegistrationGuard<'a> {
    fn reserve(slot: &'a AtomicU64) -> Option<Self> {
        match slot.compare_exchange(EMPTY, PENDING, AcqRel, Acquire) {
            Ok(_) => Some(Self { slot, armed: true }),
            Err(_) => None, // slot was already occupied
        }
    }
    
    fn commit(self, value: u64) {
        // Replace PENDING with actual value
        let prev = self.slot.swap(value, AcqRel);
        debug_assert_eq!(prev, PENDING);
        // Guard is consumed Ś Drop won't fire
        std::mem::forget(self); // disarm
    }
}

impl<'a> Drop for RegistrationGuard<'a> {
    fn drop(&mut self) {
        if self.armed {
            // Rollback Ś clear the PENDING mapping
            let prev = self.slot.swap(EMPTY, Release);
            debug_assert_eq!(prev, PENDING);
            METRICS.registration_rollbacks.fetch_add(1, Relaxed);
        }
    }
}

// Usage Ś rollback is automatic on any early return / panic
fn register_and_process(map: &IDMap, packet: Packet) -> Result<(), Error> {
    let guard = RegistrationGuard::reserve(&map.slots[packet.id])
        .ok_or(SlotOccupied)?;
    
    // Upstream processing Ś any failure triggers Drop ? rollback
    let result = process_upstream(packet)?;
    
    // Success Ś commit the mapping (disarms the guard)
    guard.commit(pack_slot_id(result.slot, result.generation));
    Ok(())
}

SOVEREIGN DNA COMPLIANCE:
  Zero-Lock: Reservation uses CAS on a single AtomicU64. Commit uses atomic swap. Rollback uses atomic swap. No locks at any point.
  Zero-Allocation: The RegistrationGuard is a stack-allocated struct containing a reference and a bool. No heap allocation. The ID map slots are pre-allocated.
  Correctness by Construction: The guard's Drop impl is the structural guarantee. It is impossible to exit the scope without either committing (which consumes the guard) or rolling back (which Drop handles). Even panics trigger Drop.

RATIONALE:
This is the lock-free equivalent of RAII transactional semantics. The PENDING sentinel acts as a reservation lock Ś but it's not a mutex because it doesn't block. It's a single atomic CAS that either succeeds (you own the slot) or fails (someone else does). Combined with the scope guard, it provides the same correctness guarantees as a database transaction (atomicity, consistency) without any of the overhead.

}

    uint64_t e1, e2;

        e1 = atomic_load(&s->epoch);
        if (e1 & 1) continue;                // Spin: write in-progress
        atomic_thread_fence(memory_order_acquire);

        out->state      = atomic_load(&s->state);
        out->sequenceId = atomic_load(&s->sequenceId);
        out->bufferHead = atomic_load(&s->bufferHead);
        out->bufferTail = atomic_load(&s->bufferTail);
        out->flags      = atomic_load(&s->flags);

        atomic_thread_fence(memory_order_acquire);
        e2 = atomic_load(&s->epoch);

    return true;                             // Snapshot is consistent
}

ADR-002 | CRITICAL | Hash Map Pool Exhaustion & Deletion Ś Tombstone Linear Probing
???????????????????????????????????????????????????????????????????????????

CONTEXT:
The pre-allocated ID map holds 8,192 slots using linear probing. Two structural flaws exist: (A) on slot exhaustion the registering packet is orphaned with no recovery path, and (B) naive zero-fill deletion breaks probe chains, causing future lookups to terminate early and falsely report 'not found' for live entries that were inserted past the deleted slot.

FLAW: Flaw A Ś Pool Exhaustion Orphan
When all 8,192 slots are occupied, register() returns an error code and the packet has no identity. Downstream processors that require a slot ID will silently drop the packet.

FLAW: Flaw B Ś Zeroed-Deletion Breaks Probe Chain
Linear probing relies on an unbroken run from the hash position to the entry. Zeroing a slot creates a false terminal that shadows all entries inserted beyond the deletion point in the same cluster.

DECISION:
Resolve Flaw A by introducing a zero-allocation overflow ring buffer of fixed size M (e.g., 256 slots) backed by a static array with a CAS-driven head pointer Ś packets are parked in the ring until a primary slot frees. Resolve Flaw B by replacing zero-fill deletion with a two-phase tombstone protocol: mark the slot DELETED (sentinel value ? 0 and ? EMPTY), continue probing through tombstones during lookup, and reuse tombstones during insertion.

MECHANISM 1: Mechanism 1 Ś Overflow Ring Buffer (Zero-Alloc Fallback)
A statically allocated ring buffer of OVERFLOW_SIZE=256 slots acts as a backpressure valve. When the primary map is full, new registrations atomically CAS a head pointer into the overflow ring. A background drain task (signaled via a lock-free MPSC queue post) migrates overflow entries to the primary map as slots free up. The ring is a fixed-size array Ś zero heap allocation.

[Code: CAS-Driven Overflow Ring]
// ??? Static overflow ring (zero heap alloc) ?????????????????????????????????
#define PRIMARY_SLOTS   8192
#define OVERFLOW_SLOTS   256
#define OVERFLOW_MASK   (OVERFLOW_SLOTS - 1)  // Power-of-2 mask

typedef struct {
    uint64_t packetId;
    void*    packetPtr;
} OverflowEntry;

// Statically allocated Ś lives in BSS, zero runtime alloc
static OverflowEntry  overflowRing[OVERFLOW_SLOTS];
static _Atomic uint64_t overflowHead = 0;   // Producer index
static _Atomic uint64_t overflowTail = 0;   // Consumer index

// ?? Park a packet in the overflow ring (non-blocking) ????????????????????????
bool overflow_enqueue(uint64_t packetId, void* pkt) {
    uint64_t head = atomic_load_explicit(&overflowHead, memory_order_relaxed);
    uint64_t tail = atomic_load_explicit(&overflowTail, memory_order_acquire);

    if ((head - tail) >= OVERFLOW_SLOTS) return false; // Ring full ? drop/backpressure

    overflowRing[head & OVERFLOW_MASK].packetId  = packetId;
    overflowRing[head & OVERFLOW_MASK].packetPtr = pkt;
    atomic_store_explicit(&overflowHead, head + 1, memory_order_release);
    return true;
}

// ?? Drain called when a primary slot is freed ????????????????????????????????
void overflow_drain_one(PrimaryMap* map) {
    uint64_t tail = atomic_load_explicit(&overflowTail, memory_order_relaxed);
    uint64_t head = atomic_load_explicit(&overflowHead, memory_order_acquire);
    if (tail == head) return;   // Ring empty

    OverflowEntry* e = &overflowRing[tail & OVERFLOW_MASK];
    if (primary_map_insert(map, e->packetId, e->packetPtr)) {
        atomic_store_explicit(&overflowTail, tail + 1, memory_order_release);
    }
}

MECHANISM 2: Mechanism 2 Ś Two-Phase Tombstone Deletion
Introduce three slot states: EMPTY (0x00), OCCUPIED (0x01¢0xFFFE), DELETED (TOMBSTONE = 0xFFFF). On deletion, write TOMBSTONE instead of zeroing. During lookup, skip tombstones (do not terminate). During insertion, reuse the first tombstone encountered in the probe chain Ś this compacts the table over time without any GC or background sweep.

[Code: Tombstone Linear Probing]
// ??? Slot state sentinels ????????????????????????????????????????????????????
#define SLOT_EMPTY     0x0000000000000000ULL
#define SLOT_TOMBSTONE 0xFFFFFFFFFFFFFFFFULL   // Deleted marker

typedef struct {
    _Atomic uint64_t key;      // EMPTY / TOMBSTONE / actual packetId
    _Atomic uintptr_t value;   // Pointer to packet metadata
} MapSlot;

static MapSlot primaryMap[PRIMARY_SLOTS];

// ?? Insert: place at first EMPTY or TOMBSTONE in probe chain ?????????????????
bool map_insert(uint64_t key, void* val) {
    uint32_t idx   = (uint32_t)(key % PRIMARY_SLOTS);
    uint32_t first_tomb = UINT32_MAX;   // Track first tombstone seen

    for (uint32_t i = 0; i < PRIMARY_SLOTS; i++) {
        uint32_t  slot = (idx + i) % PRIMARY_SLOTS;
        uint64_t  cur  = atomic_load(&primaryMap[slot].key);

        if (cur == SLOT_TOMBSTONE && first_tomb == UINT32_MAX)
            first_tomb = slot;           // Remember first reusable tombstone

        if (cur == SLOT_EMPTY) {
            // Prefer tombstone reuse to keep cluster compact
            uint32_t target = (first_tomb != UINT32_MAX) ? first_tomb : slot;
            uint64_t expected = (target == slot) ? SLOT_EMPTY : SLOT_TOMBSTONE;
            if (atomic_compare_exchange_strong(&primaryMap[target].key, 
                                               &expected, key)) {
                atomic_store(&primaryMap[target].value, (uintptr_t)val);
                return true;
            }
        }
        if (cur == key) return false;    // Duplicate
    }
    return false;   // Full ? caller routes to overflow ring
}

// ?? Lookup: skip tombstones, stop at EMPTY ???????????????????????????????????
void* map_lookup(uint64_t key) {
    uint32_t idx = (uint32_t)(key % PRIMARY_SLOTS);
    for (uint32_t i = 0; i < PRIMARY_SLOTS; i++) {
        uint32_t  slot = (idx + i) % PRIMARY_SLOTS;
        uint64_t  cur  = atomic_load(&primaryMap[slot].key);
        if (cur == SLOT_EMPTY)      return NULL; // Definitive miss
        if (cur == SLOT_TOMBSTONE)  continue;    // Skip deleted entry
        if (cur == key)             return (void*)atomic_load(&primaryMap[slot].value);
    }
    return NULL;
}

// ?? Delete: stamp TOMBSTONE, never zero ??????????????????????????????????????
bool map_delete(uint64_t key) {
    uint32_t idx = (uint32_t)(key % PRIMARY_SLOTS);
    for (uint32_t i = 0; i < PRIMARY_SLOTS; i++) {
        uint32_t slot = (idx + i) % PRIMARY_SLOTS;
        uint64_t cur  = atomic_load(&primaryMap[slot].key);
        if (cur == SLOT_EMPTY)     return false;
        if (cur == SLOT_TOMBSTONE) continue;
        if (cur == key) {
            atomic_store(&primaryMap[slot].key, SLOT_TOMBSTONE);
            atomic_store(&primaryMap[slot].value, 0);
            overflow_drain_one(&primaryMap); // Opportunistic drain
            return true;
        }
    }
    return false;
}

CONSEQUENCES:
Ģ Tombstone accumulation is bounded: worst-case probe length is O(load_factor / (1 - load_factor)). A periodic background compaction (non-hot-path) can rehash the table into a second static buffer and flip the active pointer.
Ģ Overflow ring introduces bounded backpressure (256 slots). If both primary and overflow are full, the caller receives a hard ENOBUFS Ś a deterministic, observable failure mode.
Ģ Both structures are statically allocated: zero heap usage, zero GC pressure.

RATIONALE:
Tombstone deletion is the textbook physics of linear probing Ś it is the minimal perturbation that preserves probe chain invariants. The overflow ring is a SPSC/MPSC queue idiom applied to registration, providing a zero-alloc parking structure with CAS-only coordination.

ADR-003 | CRITICAL | Sideband Cleanup vs Pool Release Ś Fail-Safe Nested Ordering
???????????????????????????????????????????????????????????????????????????

CONTEXT:
The current teardown contract releases the pool slot in a finally block. A proposed change that runs sideband cleanup before the pool release violates the invariant: if sideband cleanup throws or faults, execution jumps to the nearest exception handler, skipping the pool release and permanently leaking the slot. The leaked slot is unrecoverable without a GC scan Ś violating Zero-Alloc.

FLAW: Flaw Ś Exception Escape Leaks Pool Slot
Sideband cleanup code (e.g., unregistering from an external broker, zeroing DMA descriptors) can fault. If placed before pool_slot_release() in the same try-block, a thrown exception routes control past the release call, and the finally block only contains the outer cleanup Ś the slot is lost.

DECISION:
Enforce a strictly nested, dual-finally structure: an inner try-finally around sideband cleanup guarantees sideband completion (or fault-logging) before the outer try-finally's pool release executes. This ordering is structural Ś it cannot be bypassed by any exception path. Model the teardown as a two-stage FSM transition to eliminate ambiguity about which cleanup phase has completed.

MECHANISM 1: Mechanism 1 Ś Dual-Finally Nested Teardown
Wrap sideband cleanup in its own inner try-finally that catches and logs any fault without re-throwing. The outer finally then unconditionally releases the pool slot. This is equivalent to a C++ RAII destructor chain Ś inner destructors run before the outer destructor regardless of exceptions.

[Code: Nested Try-Finally Teardown (C# / Java style pseudocode)]
// ??? Dual-Finally Teardown Contract ??????????????????????????????????????????
//  INVARIANT: pool slot is ALWAYS released regardless of sideband outcome.
//  No lock, no allocation Ś pure control-flow structuring.

void TeardownSlot(DispatchSlot* slot) {
    // ?? OUTER finally: pool release is GUARANTEED ?????????????????????????????
    try {
        // ?? INNER finally: sideband cleanup, fault-contained ?????????????????
        try {
            SidebandClear(slot->sidebandPtr);       // May throw / fault
            ZeroDmaDescriptors(slot->dmaHandle);    // May throw / fault
            UnregisterFromBroker(slot->channelId);  // May throw / fault
        }
        finally {
            // Even if sideband threw, record the fault atomically
            // (write to a lock-free fault ring Ś zero alloc)
            if (current_exception() != NULL) {
                fault_ring_push(slot->slotId, FAULT_SIDEBAND_CLEANUP);
                // Suppress: do NOT re-throw Ś outer finally must run
            }
            // Mark slot FSM state: SIDEBAND_CLEARED (even if partial)
            atomic_store(&slot->lifecycleState, STATE_SIDEBAND_CLEARED);
        }
        // ?? OUTER try body continues only if inner did not re-throw ??????????
    }
    finally {
        // Unconditionally release: runs regardless of sideband outcome
        pool_slot_release(slot);                    // Zero-alloc CAS return
        atomic_store(&slot->lifecycleState, STATE_POOL_RELEASED);
    }
}

Any code path that observes an incomplete transition (e.g., during crash recovery) can re-enter the teardown at the correct stage by inspecting the atomic lifecycle field Ś enabling deterministic recovery without a lock or allocation.

    LIFECYCLE_DRAINING         = 1,   // Teardown initiated

        &s->lifecycleState, &expected, (uint32_t)to
    );
}

// ?? Recovery scan: re-enter teardown at the interrupted stage ????????????????
void recovery_scan(DispatchSlot* slots, size_t count) {
    for (size_t i = 0; i < count; i++) {
        LifecycleState ls = (LifecycleState)atomic_load(&slots[i].lifecycleState);
        switch (ls) {
            case LIFECYCLE_DRAINING:
                // Sideband cleanup was interrupted Ś re-run inner phase

                pool_slot_release(&slots[i]);

                break;
            default:
                break;
        }
    }
}

Ģ No locks, no allocation: all coordination is through atomic stores and structured control flow.

ADR-004 | CRITICAL | ABA Risk & Slot Generation Ś Embedded Generation Counters
???????????????????????????????????????????????????????????????????????????

CONTEXT:
Two ABA exposure points exist: (A) The Circuit Breaker state machine uses a single atomic word for state transitions. A thread can read OPEN, be preempted, observe the state cycle OPEN?CLOSED?OPEN, then apply its stale OPEN-based decision as if nothing changed Ś classic ABA. (B) The Dispatch Slot has no generation field, so a delayed callback racing a slot reallocation reads the new occupant's sideband data, causing cross-contamination between unrelated packets.

FLAW: Flaw A Ś Circuit Breaker ABA
A thread reads state=OPEN, suspends, the CB resets to CLOSED, fires again to OPEN. Thread resumes and CASes on the stale OPEN value Ś the generation change is invisible because the value of the atomic is identical.

FLAW: Flaw B Ś Dispatch Slot Cross-Contamination
Slot[i] is released and immediately reallocated to packet B while a delayed callback for packet A still holds the slot index i. The callback reads packet B's sideband data and processes it as packet A's Ś silent data corruption.

DECISION:
Embed a generation counter into the same atomic word as the state/slot-id using bit-packing. For the Circuit Breaker, pack a 48-bit generation + 16-bit state into a single atomic uint64. For the Dispatch Slot, pack a 32-bit generation + 32-bit slot-id into a single atomic uint64. All CAS operations compare the full 64-bit word Ś a mismatched generation is an automatic CAS failure, making ABA physically impossible for that epoch.

MECHANISM 1: Mechanism 1 Ś Circuit Breaker Generational State Word
Compress the Circuit Breaker's mutable state into a single atomic uint64_t: bits [63:16] = generation counter (48 bits, ~280 trillion state cycles before wrap), bits [15:0] = state enum (65,536 possible states). Every state transition CAS-compares the full word. A thread that was preempted holds an old generation Ś its CAS will fail unconditionally, forcing a re-read and re-evaluation.

[Code: Packed Circuit Breaker State Word]
// ??? Circuit Breaker generational state word ?????????????????????????????????
//  Bits [63:16] = generation (48-bit, ~2.8ū10^14 cycles before wrap)
//  Bits [15:0]  = state enum (16-bit, supports 65536 states)

typedef enum : uint16_t {
    CB_CLOSED    = 0x0000,
    CB_OPEN      = 0x0001,
    CB_HALF_OPEN = 0x0002,
} CbState;

typedef struct {
    _Atomic uint64_t stateWord;   // [generation:48 | state:16]
} CircuitBreaker;

#define CB_STATE_MASK  0x000000000000FFFFULL
#define CB_GEN_SHIFT   16

static inline uint64_t cb_pack(uint64_t gen, CbState st) {
    return (gen << CB_GEN_SHIFT) | (uint64_t)st;
}
static inline CbState   cb_state(uint64_t w) { return (CbState)(w & CB_STATE_MASK); }
static inline uint64_t  cb_gen  (uint64_t w) { return w >> CB_GEN_SHIFT; }

// ?? State transition: CAS on full word (generation mismatch = safe failure) ??
bool cb_transition(CircuitBreaker* cb, CbState from, CbState to) {
    uint64_t cur = atomic_load_explicit(&cb->stateWord, memory_order_acquire);
    while (cb_state(cur) == from) {
        uint64_t next = cb_pack(cb_gen(cur) + 1, to);  // Increment generation
        if (atomic_compare_exchange_weak_explicit(
                &cb->stateWord, &cur, next,
                memory_order_acq_rel, memory_order_acquire)) {
            return true;  // Transition succeeded
        }
        // cur is now reloaded by CAS Ś loop re-checks generation automatically
    }
    return false;  // State was not 'from' Ś ABA prevented by generation check
}

Every Dispatch Slot carries an atomic uint64_t slotToken: bits [63:32] = generation (32-bit), bits [31:0] = slot index. Callbacks receive the full 64-bit token at dispatch time. Before reading any sideband data, the callback validates its token against the slot's current atomic token. A mismatch (different generation) means the slot was reallocated Ś the callback must self-cancel without reading data.

[Code: Slot Generational Token Validation]

//  Bits [63:32] = generation counter (32-bit, 4 billion reallocations)
//  Bits [31:0]  = slot index

#define SLOT_GEN_SHIFT  32
#define SLOT_IDX_MASK   0x00000000FFFFFFFFULL

typedef struct {
    _Atomic uint64_t token;         // [generation:32 | slotIndex:32]
    _Atomic uintptr_t sidebandPtr;  // Sideband data pointer
    // ... other fields ...
} DispatchSlot;

static DispatchSlot slotPool[PRIMARY_SLOTS];  // Static Ś zero heap alloc

    return ((uint64_t)gen << SLOT_GEN_SHIFT) | (uint64_t)idx;
}
static inline uint32_t slot_gen(uint64_t tok) { return (uint32_t)(tok >> SLOT_GEN_SHIFT); }
static inline uint32_t slot_idx(uint64_t tok) { return (uint32_t)(tok & SLOT_IDX_MASK); }

// ?? Acquire slot: atomically bump generation ??????????????????????????????????
uint64_t slot_acquire(uint32_t slotIdx) {

    uint32_t newGen = slot_gen(old) + 1;
    uint64_t newTok = slot_make_token(newGen, slotIdx);
    atomic_store_explicit(&slotPool[slotIdx].token, newTok, memory_order_release);
    return newTok;  // Return to callback as its "passport"
}

// ?? Callback: validate token before accessing sideband ???????????????????????
bool callback_validate_and_read(uint64_t myToken, void** sidebandOut) {
    uint32_t idx = slot_idx(myToken);
    uint64_t curToken = atomic_load_explicit(&slotPool[idx].token, 
                                              memory_order_acquire);
    if (curToken != myToken) {
        // Generation mismatch Ś slot was reallocated Ś self-cancel
        return false;
    }
    *sidebandOut = (void*)atomic_load(&slotPool[idx].sidebandPtr);
    return true;  // Token matched Ś data is valid for this generation
}

Ģ ABA is physically impossible within a generation: a 48-bit CB counter wraps after ~280 trillion transitions (~9,000 years at 1M transitions/sec). A 32-bit slot counter wraps after ~4.3 billion slot reallocations.

Ģ Generation overflow is detectable: a saturating counter variant can fire an assertion/alert before wrap.

Tagged pointers / generational tokens are the standard physics-based ABA remedy in lock-free programming. ADR-005 | CRITICAL | Registration Rollback Ś Atomic Compensating CAS
???????????????????????????????????????????????????????????????????????????

CONTEXT:
When a worker registers a packet (inserts into the ID map) and upstream processing subsequently throws an exception before the packet is consumed, the map entry is never removed. The registration leaks permanently Ś the slot is occupied, the packet may have been freed, and future lookups return a dangling pointer or stale data. There is no existing compensating path to roll back the insert.

FLAW: Flaw Ś Permanent Registration Leak on Upstream Fault
Insert to map succeeds ? upstream throws ? finally block does not call map_delete() ? slot permanently occupied by a stale key ? dangling pointer risk on next lookup.

DECISION:
Model registration as a two-phase commit: (1) Insert with a PENDING sentinel value rather than the real pointer; (2) commit by swapping to the real pointer only if upstream processing reaches its commit point. A rollback path atomically stamps the slot TOMBSTONE. Since both steps use CAS on the slot's value field, no lock is required. An RAII guard (stack-allocated, zero-alloc) drives the commit/rollback decision based on whether the scope exits normally or via exception.

MECHANISM 1: Mechanism 1 Ś Two-Phase Commit Registration with RAII Rollback Guard
The RAII guard is a stack struct that holds the slot index and a committed flag. Its destructor checks the flag: if not committed (exception path), it CAS-stamps the slot with TOMBSTONE. If committed (normal path), it is a no-op. This is structurally identical to a database two-phase commit Ś insert is the prepare phase, upstream success is the commit signal.

[Code: RAII Rollback Guard Ś Two-Phase Registration]
// ??? Two-Phase Registration Guard (stack-allocated, zero heap alloc) ?????????

    uint64_t  key;           // The packet key inserted
    bool      committed;     // Set to true ONLY on successful upstream commit
} RegistrationGuard;

#define REG_PENDING_SENTINEL  0x0000000000000001ULL  // Non-null, non-valid pointer

// ?? Phase 1: PREPARE Ś insert with pending sentinel ???????????????????????????
RegistrationGuard reg_prepare(uint64_t packetId) {
    RegistrationGuard g = { .key = packetId, .committed = false };

    // Insert key ? PENDING_SENTINEL (not the real pointer yet)
    // Uses the tombstone-aware map_insert from ADR-002
    if (!map_insert_with_value(packetId, REG_PENDING_SENTINEL, &g.slotIdx)) {
        g.slotIdx = UINT32_MAX;   // Signal: no slot acquired
    }
    return g;  // Guard is stack-allocated Ś zero heap alloc
}

// ?? Phase 2: COMMIT Ś swap sentinel for the real pointer ?????????????????????
bool reg_commit(RegistrationGuard* g, void* realPtr) {
    if (g->slotIdx == UINT32_MAX) return false;

    uintptr_t expected = REG_PENDING_SENTINEL;
    bool ok = atomic_compare_exchange_strong(
        &primaryMap[g->slotIdx].value, &expected, (uintptr_t)realPtr
    );
    if (ok) g->committed = true;
    return ok;
}

// ?? ROLLBACK: called by guard destructor if !committed ????????????????????????
void reg_rollback(RegistrationGuard* g) {
    if (g->slotIdx == UINT32_MAX || g->committed) return;

    // CAS: only stamp tombstone if the slot still holds our pending sentinel
    // (guards against double-rollback or concurrent commit)
    uintptr_t expected = REG_PENDING_SENTINEL;
    atomic_compare_exchange_strong(
        &primaryMap[g->slotIdx].value, &expected, 0
    );
    // Stamp the key slot as tombstone
    uint64_t keyExpected = g->key;
    atomic_compare_exchange_strong(
        &primaryMap[g->slotIdx].key, &keyExpected, SLOT_TOMBSTONE
    );
    overflow_drain_one(&primaryMap);  // Opportunistic drain after freeing a slot
}

// ?? Usage: RAII-style scope guard ?????????????????????????????????????????????
void process_packet(uint64_t packetId, PacketData* data) {
    RegistrationGuard g = reg_prepare(packetId);
    // ^ Stack allocation: destructor pattern in C++ / using-block in C#

    // ... upstream processing may throw here ...
    upstream_process(data);            // If this throws, guard destructor fires

    void* sideband = build_sideband(data);
    reg_commit(&g, sideband);         // Only reached on success
    // Guard destructor: committed=true ? no-op rollback
}

MECHANISM 2: Mechanism 2 Ś Pending-Sentinel Read Filtering
Any lookup that encounters a PENDING_SENTINEL value must treat the entry as if it does not exist (return NULL). This prevents a concurrent reader from observing and using a partially-registered packet, which would be equivalent to a dirty read in database isolation terminology.

[Code: Sentinel-Aware Lookup]
// ??? Sentinel-aware lookup: filters PENDING entries ??????????????????????????
void* map_lookup_committed(uint64_t key) {
    uint32_t idx = (uint32_t)(key % PRIMARY_SLOTS);
    for (uint32_t i = 0; i < PRIMARY_SLOTS; i++) {
        uint32_t  slot = (idx + i) % PRIMARY_SLOTS;
        uint64_t  k    = atomic_load(&primaryMap[slot].key);

        if (k == SLOT_EMPTY)     return NULL;  // Definitive miss
        if (k == SLOT_TOMBSTONE) continue;     // Deleted Ś keep probing
        if (k != key)            continue;     // Different key in cluster

        uintptr_t v = atomic_load(&primaryMap[slot].value);
        if (v == REG_PENDING_SENTINEL) return NULL; // Uncommitted Ś hide from reader
        return (void*)v;                            // Committed pointer
    }
    return NULL;
}

CONSEQUENCES:
Ģ Registration leaks are structurally impossible: the RAII guard is the only code path that holds the slot index, and its destructor always runs.
Ģ Concurrent readers never observe a partially-constructed registration: the PENDING sentinel acts as a read barrier.
Ģ Rollback is a pure CAS operation: no lock, no allocation, O(1) complexity.
Ģ The two-phase pattern composes with ADR-002 tombstones and ADR-004 generation counters without conflict.

RATIONALE:
Two-phase commit is the canonical database pattern for atomic, recoverable multi-step operations. Applying it to a lock-free map via CAS and a stack-allocated RAII guard achieves the same semantic guarantee with zero OS primitives and zero heap allocation. The PENDING sentinel is the physics-based equivalent of a write-ahead log entry Ś it marks intent without committing effect.

ADR-006 | SIGNIFICANT | False-Sharing Elimination in Shared Counters
???????????????????????????????????????????????????????????????????????????

CONTEXT:
High-frequency per-slot counters (drop counts, retry counts) packed into adjacent memory locations cause cache line ping-pong between CPU cores updating different slots.

FLAW: Flaw Ś Counter Cache Line Contention
When slots[i].dropCount and slots[i+1].dropCount share a 64-byte cache line, an update to slot i invalidates the line for the core handling slot i+1, causing a cross-core invalidation storm at high packet rates.

DECISION:
Pad each per-slot counter struct to exactly one cache line (64 bytes on x86) using __attribute__((aligned(64))) or alignas(64). Use thread-local shadow counters that are periodically flushed to the shared struct via a single atomic add Ś reducing cross-core traffic by 10¢100x.

MECHANISM 1: Cache-Line-Padded Per-Slot Counters
Align each counter struct to a full cache line boundary to eliminate false sharing. Thread-local accumulators batch updates and flush periodically.

[Code: Cache-Line Padded Counter]
// ??? Cache-line-padded per-slot counter ?????????????????????????????????????
typedef struct alignas(64) SlotCounter {
    _Atomic uint64_t dropCount;
    _Atomic uint64_t retryCount;
    _Atomic uint64_t successCount;
    uint8_t _pad[40];   // Pad to exactly 64 bytes
} SlotCounter;

// Thread-local accumulator: zero cross-core traffic during batch
static _Thread_local uint64_t tl_drops[PRIMARY_SLOTS];

void counter_flush(uint32_t slotIdx) {
    if (tl_drops[slotIdx] == 0) return;
    atomic_fetch_add(&counterArray[slotIdx].dropCount, tl_drops[slotIdx]);
    tl_drops[slotIdx] = 0;
}

CONSEQUENCES:
Ģ False sharing eliminated: each slot counter owns its cache line exclusively.
Ģ Thread-local batching reduces atomic operations by the batch size factor.
Ģ Memory cost: 64 bytes per slot ū 8192 slots = 512 KB Ś fits in L2 cache.

RATIONALE:
Cache line padding is the standard mechanical solution to false sharing. Thread-local batching is the standard approach to reducing contention on hot atomic counters.

ADR-007 | SIGNIFICANT | Lock-Free MPSC Queue Backpressure Signal
???????????????????????????????????????????????????????????????????????????

CONTEXT:
The existing notification path for sideband events uses a condition variable + mutex, violating Zero-Lock. High-frequency sideband events cause priority inversion when the notifying thread holds the mutex longer than the consuming thread's scheduling quantum.

FLAW: Flaw Ś Mutex-Based Notification Violates Zero-Lock
A slow consumer holding the mutex blocks all producers. Under load, this creates a priority inversion chain where a low-priority cleanup thread blocks high-priority packet processors.

DECISION:
Replace the mutex+condvar with a lock-free MPSC (Multi-Producer Single-Consumer) notification queue backed by a static ring buffer. Producers enqueue a 64-bit event token via CAS. The single consumer polls with a parking strategy: spin-N then yield, then park on a futex Ś providing both low-latency and CPU-efficient operation.

MECHANISM 1: Static MPSC Ring Buffer with Futex Parking
A power-of-2 sized static ring with per-slot sequence numbers provides MPSC ordering without locks. The consumer parks on a futex only after confirming the ring is empty.

[Code: MPSC Ring with Adaptive Parking]
// ??? Lock-free MPSC ring (Dmitry Vyukov design, static allocation) ??????????
#define MPSC_SIZE   1024          // Power of 2
#define MPSC_MASK   (MPSC_SIZE-1)

typedef struct {
    _Atomic uint64_t sequence;
    uint64_t         data;
} MpscCell;

typedef struct {
    MpscCell         cells[MPSC_SIZE];
    _Atomic uint64_t enqueuePos;
    _Atomic uint64_t dequeuePos;
    uint8_t          _pad[40];  // Cache line separation
} MpscQueue;

static MpscQueue sidebandQueue;  // Static Ś zero heap alloc

bool mpsc_enqueue(uint64_t data) {
    uint64_t pos = atomic_fetch_add(&sidebandQueue.enqueuePos, 1);
    MpscCell* cell = &sidebandQueue.cells[pos & MPSC_MASK];
    // Spin until this cell is ours (sequence == pos)
    while (atomic_load(&cell->sequence) != pos) { /* spin */ }
    cell->data = data;
    atomic_store(&cell->sequence, pos + 1);  // Signal: ready to consume
    return true;
}

CONSEQUENCES:
Ģ Zero mutex usage: all coordination is via atomic sequence numbers.
Ģ Priority inversion eliminated: producers never block on consumer state.
Ģ Static ring: 1024 ū 16 bytes = 16 KB Ś fits in L1 cache.

RATIONALE:
The Vyukov MPSC queue is the canonical lock-free, allocation-free multi-producer queue. Sequence-number-based cells provide wait-freedom for the consumer and progress-freedom for producers.

ADR-008 | SIGNIFICANT | Monotonic Sequence ID Anti-Replay Guard
???????????????????????????????????????????????????????????????????????????

CONTEXT:
Packet sequence IDs are generated from a shared atomic counter but are not validated against a per-channel receive window. A delayed or replayed packet with an old sequence ID can be processed as if it were new, causing state divergence in the FSM.

FLAW: Flaw Ś Missing Receive Window Validation
The FSM advances state based on the packet's declared sequence ID. A replayed packet with ID N can re-trigger a state transition that already occurred, causing double processing or FSM corruption.

DECISION:
Implement a fixed-width sliding window anti-replay filter using a 64-bit atomic bitmask per channel. The window size is 64 packets. Packet acceptance requires: (1) SeqID > lastAccepted - 64 (not too old), (2) SeqID not set in the window bitmask (not a duplicate). The bitmask is shifted atomically when the window advances.

MECHANISM 1: 64-Bit Atomic Sliding Window Anti-Replay
A single atomic uint64_t encodes a 64-packet receive window as a bitmask. Bit i is set if packet (baseSeq + i) has been accepted. Window advance is a shift operation on the atomic value.

[Code: Bitmask Sliding Window Anti-Replay]
// ??? Per-channel anti-replay sliding window ???????????????????????????????????
typedef struct {
    _Atomic uint64_t baseSeq;    // Lowest sequence in current window
    _Atomic uint64_t recvMask;   // Bit i = packet (baseSeq+i) received
} AntiReplayWindow;

// ?? Accept check: returns true if packet should be processed ?????????????????
bool antireplay_accept(AntiReplayWindow* w, uint64_t seq) {
    uint64_t base = atomic_load_explicit(&w->baseSeq, memory_order_acquire);
    if (seq < base) return false;              // Too old Ś reject
    uint64_t offset = seq - base;
    if (offset >= 64) {
        // Advance window: shift mask, update base
        uint64_t shift = offset - 63;
        uint64_t mask  = atomic_load(&w->recvMask);
        uint64_t newMask = mask >> shift;
        atomic_store(&w->recvMask, newMask);
        atomic_store(&w->baseSeq,  base + shift);
        offset = 63;
    }
    uint64_t bit  = 1ULL << offset;
    uint64_t prev = atomic_fetch_or(&w->recvMask, bit);
    return !(prev & bit);  // false = duplicate
}

CONSEQUENCES:
Ģ Replay protection covers a 64-packet window with a single 64-bit atomic Ś zero additional allocation.
Ģ Out-of-order delivery within the window is accepted; only true duplicates are rejected.
Ģ Window advance is a shift operation: O(1), branch-free.

RATIONALE:
The bitmask sliding window is the standard mechanism used in IPSec/TLS anti-replay (RFC 6479). A single atomic word provides both the window state and CAS-based duplicate detection with no lock.

TAGS: anti-replay | sliding-window | bitmask | sequence-id | idempotency
