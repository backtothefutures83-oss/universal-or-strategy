Greptile Summary
This PR attempts a pure-style pass over 66 C# source files using an automated Python script to add curly braces around 1,218 single-line control statements (Codacy IDE0011). While the majority of simple single-statement bodies were wrapped correctly, the script has three categories of structural mis-handling that introduce hard compilation errors in at minimum 4–6 files.
* Broken multi-line method calls: wherever the body of a braceless if/else was a call whose arguments span multiple lines (e.g. Print(\n string.Format(...)\n)), the script inserted { after the opening line and } on the next line — enclosing only the opening ( inside the block and leaving all arguments and the closing ) as orphaned syntax outside it. Confirmed in V12_002.SIMA.Lifecycle.cs (lines 87–93, 352–358) and V12_002.UI.IPC.Commands.Fleet.cs (lines 164–180).
* Braces injected inside multi-line if conditions: when a condition spanned multiple lines starting with if (\n expr\n && expr\n), the script treated the first operand as the body, placing { and } inside the parenthesized condition — not after it. Confirmed in V12_002.SIMA.Lifecycle.cs (~lines 283, 390, 825), V12_002.UI.IPC.Commands.Fleet.cs (lines 253, 268, 310, 373), and V12_002.Orders.Callbacks.Propagation.cs (lines 54, 137, 163, 338, 351).
* Lambda bodies orphaned outside if blocks: for Click += (s, e) =>\n{ ... } patterns, the script treated the line ending with => as the single-statement body, placing } after => and leaving the lambda body { ... } outside the if scope as a standalone block. Confirmed at ~10 sites in V12_002.UI.Panel.Handlers.cs.
Confidence Score: 1/5
Do not merge — the automated brace-insertion script has introduced hard syntax errors across at least four files, making the build non-functional.
