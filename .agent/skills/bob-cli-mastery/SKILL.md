# Bob CLI Mastery

## Overview
Bob CLI handles V12 refactoring work via two distinct workflows:
1. **Epic Planning** (`/v12-epic-planner` mode) -- Four-phase planning pipeline that produces ticket files.
2. **Ticket Execution** (`/v12-engineer` mode) -- One ticket per isolated session, PLAN-THEN-EXECUTE.

---

## Entry Points -- Choose One

| Mode | Command | When to Use |
|------|---------|-------------|
| **YOLO (Recommended)** | `/epic-run <slug> <desc>` in Orchestrator mode | Full automation -- planning + execution in one session. Only manual step: F5 in NinjaTrader. |
| **Manual phases** | `/epic-intake` → `/epic-plan` → `/epic-validate` → `/epic-tickets` | Use when you want full control at each phase, or when recovering from a mid-epic failure. |
| **Single ticket** | `/ticket <path>` in v12-engineer mode | Use to re-run or fix a specific ticket after YOLO completes. |

**First time on a new epic: use manual phases to understand the flow. After that: `/epic-run`.**

## Epic Workflow (Traycer-Parity)

The Epic Workflow replicates Traycer's 6-phase refactoring pipeline in Bob. The key difference:
- **Traycer:** One continuous session handles planning AND execution via agent handoffs.
- **Bob:** Epic session = planning only (phases 1-4). Each ticket = its own new isolated session.

### Phase Map

| Phase | Command | Mode | Output | Gate |
|-------|---------|------|--------|------|
| 1 | `/epic-intake <slug> <description>` | v12-epic-planner | `docs/brain/[epic]/00-scope.md` | Director confirms scope |
| 2 | `/epic-plan <slug>` | v12-epic-planner | `01-analysis.md` + `02-approach.md` | Director types APPROVED |
| 3 | `/epic-validate <slug>` | v12-epic-planner | Updated 01/02 in-place | Director signs off |
| 4 | `/epic-tickets <slug>` | v12-epic-planner | `ticket-XX-*.md` + `EXECUTION_GUIDE.md` | Director approves breakdown |
| 5 | `/ticket docs/brain/[epic]/ticket-XX.md` | v12-engineer | src/ edits | Director presses F5 |

### Starting an Epic Session in Bob
1. Open Bob, switch to `/v12-epic-planner` mode
2. Type `/epic-intake <epic-slug> <target description>`
3. Follow the gates -- confirm at each STOP before advancing

### Executing Tickets
For EACH ticket (after epic planning is done):
1. Open a NEW Bob session
2. Switch to `/v12-engineer` mode
3. Type `/ticket docs/brain/[epic-slug]/ticket-XX-[name].md`
4. Await `[TICKET-GATE]` -- type APPROVED after reviewing the plan
5. Await `[TICKET-COMPLETE]` -- run manual director gates below

## Ticket Execution Header Prompt Trick

To force Bob into the V12 Plan-then-Execute protocol, pre-load context with a strict header
BEFORE pasting anything, especially for raw tickets (not generated by /epic-tickets):

```markdown
MISSION: Phase 7 Complexity Extraction Epic -- V12 Photon Kernel
BUILD_TAG_BASELINE: [PREVIOUS_TAG]
REPO: c:\WSGTA\universal-or-strategy
BRANCH: feature/phase7-sprint5-extraction
SPEC REF: docs/brain/[epic-slug]/02-approach.md

Execute PLAN-THEN-EXECUTE PROTOCOL. Produce a written plan with helper
names, signatures, and caller impact. STOP and confirm before coding.
Post-edit: run deploy-sync.ps1 + complexity_audit.py + bump BUILD_TAG.

--- TICKET BELOW ---
[paste ticket]
```

## Director Manual Gates (Post-Ticket Checklist)

Because Bob's P6 Verifier runs in Ask mode and CANNOT run shell commands, the Director MUST manually run:
1. `powershell -File .\deploy-sync.ps1` (Hardlinks + ASCII gate)
2. `python scripts/complexity_audit.py` (CYC floor verification)
3. `python scripts/check_ascii.py` (ASCII gate)
4. `grep -c "lock\s*(" src/*.cs` (Zero-lock gate)
5. **NinjaTrader F5** (Compile + BUILD_TAG verification)

## Red Flags -- STOP and Start Over

- **Missing epic-intake scope doc**: Jumping straight to /epic-plan without 00-scope.md.
- **Skipping a gate**: Proceeding from intake to plan without Director confirmation.
- **Epic planner editing src/**: The v12-epic-planner mode is restricted to docs/ only. If it attempts src/ edits, halt.
- **Ticket not self-contained**: A ticket that references "the approach from before" instead of explicit paths is not ready.
- **P6 inference trusted implicitly**: "Bob said PASS, so the ASCII gate is fine." -- run the gate manually.
- **Missing Header on raw tickets**: Dropping a raw ticket into Bob without the BUILD_TAG_BASELINE header.
- **Task Splitting during P4**: Allowing the Red Team to audit only part of the plan.

## Common Mistakes

| Excuse | Reality |
|--------|---------|
| "Bob's verification report says PASS for grep." | P6 runs in Ask mode and cannot execute grep. It inferred the result. Run it manually. |
| "I'll skip /epic-validate, the plan looks fine." | Validation catches invariant gaps and over-complex approaches. It is not optional. |
| "I'll just paste the ticket, Bob knows what to do." | Without the formal Header Prompt or /ticket command, Bob drops V12 DNA constraints. |
| "The deploy-sync wasn't needed, the diff was small." | Editing a file breaks the hardlink inode. deploy-sync.ps1 is MANDATORY for F5 compilation. |
| "I can run multiple tickets in one Bob session." | Each ticket requires a fresh isolated session to prevent context drift and scope bleed. |

## Bob Code Review Feature (Built-in)

Bob provides AI-powered code reviews to catch errors early and validate issue coverage.

### Review Modes
1. **Branch Comparison**: `/review <branch>` (Compare uncommitted changes against a target branch).
2. **Issue Coverage**: `/review <issue-url> --issue-coverage` (Validate changes against GitHub issue requirements).
3. **Local Review**: `/review` (Analyze uncommitted changes in the working directory).

### The Review Workflow
1. **Initiate**: Use the sidebar Review Panel or the `/review` slash command.
2. **Analysis**: Bob analyzes changes for errors, quality, and issue alignment.
3. **Findings**: Findings appear in the **Bob Findings** panel.
    - **Open**: Needs attention.
    - **In Progress**: Bob is fixing the issue.
    - **Resolved**: Addressed.
4. **Action**: Use "Fix with Bob" to automate repairs or "Mark as Resolved" for manual fixes.

### Best Practices for V12
- **Run before commit**: Always run `/review` before committing to catch DNA violations.
- **Issue Validation**: Use `--issue-coverage` when working on specific Sprint tickets to ensure all logic gaps are closed.
- **Configurable Exclusions**: Use glob patterns in Bob Settings > Bob Findings to exclude `.bak` or `.log` files.

## Red Flags - STOP and Start Over

- **P6 inference trusted implicitly**: "Bob said PASS, so the ASCII gate is fine." 
- **Adjudicator looping**: Sending Adjudicator clarifications back to the Architect instead of resolving them against established Epic DNA.
